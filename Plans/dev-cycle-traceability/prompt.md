# Feature: Dev Cycle Full Traceability

## Problem Statement

Currently, when agents raise bugs, features, or tickets during a development cycle, there is no structured link back to the parent work item (FR or bug) that the cycle is processing. Additionally:

1. **Bugs raised during cycles** (e.g., deployment failures from CI/CD simulation) have no `related_work_item_id` — you cannot trace a deployment-failure bug back to the FR/bug that triggered the cycle.
2. **Tickets** within a cycle have no explicit reference to the parent FR/bug — only an indirect link via `cycle_id → cycle.work_item_id`.
3. **Development cycles** show the pipeline team via `pipeline_run.team`, but there is no structured log of **team feedback** (QA rejections, security findings, design critique) attached to the work item or its tickets.
4. **Completed features** in the Feature Browser have no traceability report — when a cycle completes and produces a Feature record, the traceability report generated by `traceability-reporter` is not persisted or linked.
5. **Ticket descriptions** created by agents lack structured fields for "issue analysis" and "considered fixes" — they're free-text only.

## Desired Behavior

1. **Work item reference on all child artifacts**: Every bug, ticket, or feature created during a dev cycle must carry the parent work item reference (`related_work_item_id` + `related_work_item_type`).
2. **Structured ticket fields**: Tickets gain `issue_description` (detailed problem analysis) and `considered_fixes` (JSON array of fix options with rationale) fields alongside the existing `description`.
3. **Team feedback log**: A new `cycle_feedback` table captures feedback from pipeline teams (QA rejections, security findings, design critique) linked to the cycle and optionally to a specific ticket. This makes team feedback queryable and traceable.
4. **Cycle team display**: The cycle detail response includes the team name and a summary of all feedback entries, making it clear which team(s) worked on it and what they found.
5. **Traceability report on features**: When a cycle completes, the traceability report (FR coverage, test mapping) is stored on the Feature record. The `features` table gains a `traceability_report` TEXT column.
6. **Bug source tracing**: Bugs auto-created by `completeCycle()` (deployment failures) include `related_work_item_id` pointing to the cycle's work item, and `related_cycle_id` pointing to the cycle.

## Scope

- Schema changes: Add columns to `bugs`, `tickets`, `features`; new `cycle_feedback` table
- Service layer: Modify bug, ticket, cycle, feature, and pipeline services
- API: New endpoints for cycle feedback; modified create/response shapes for bugs, tickets, features
- Frontend: Display feedback log on CycleView; show traceability report on Feature detail; show related work item links
- Full backwards compatibility: existing records without new fields continue to work
