1. Executive Summary

Grade D — High Risk. The application operates in a fully unauthenticated state: every API endpoint, including all workflow state-transition operations, is publicly accessible with no credentials required. Compliance pass rate is 8.3% against OWASP-ASVS L2 (2 of 19 controls pass) and 0% against SOC2-Type2 full requirements. The application must not be exposed to untrusted network access in its current state.

Top 3 Risks

  • [MERGED-001] Zero Authentication on All API Endpoints (High) All 18+ API endpoints — including workflow state transitions (approve, reject, dispatch, route), intake webhooks, and the metrics endpoint — are fully accessible to any unauthenticated HTTP client. This is the root amplifier for every other finding. OWASP A01 / CWE-306 / ASVS V2 / SOC2 CC6.1.
  • [PEN-002] Fast-Track Route Override Bypasses Entire Assessment Pod (High) POST /api/work-items/:id/route accepts {"overrideRoute":"fast-track"} from any unauthenticated caller, collapsing the entire assessment pipeline (backlog → approved) in a single request. The four-role assessment pod is completely bypassed. Items can be dispatched to teams without any quality review.
  • [COMP-002] No TLS — All Data Transmitted in Cleartext (High) Nginx binds HTTP-only. All API traffic, including any future credential payloads and work item data, is transmitted unencrypted. This fails SOC2 CC6.1 and all OWASP-ASVS data-protection controls.

2. Consolidated Findings

Severity definitions: Critical = RED-ID confirmed live breach; High = PEN-ID theoretical exploit chain with clear impact; Medium = SAST/COMP gap with exploitability conditions; Low = best-practice / configuration issue. Phase 2 produced no RED-ID findings; all findings are Theoretical.

MERGED-001 Zero Authentication on All API Endpoints
High Theoretical PEN-001 + SAST-001 + COMP-001
Sources: pen-tester · static-analyzer · compliance-auditor CWE-306 · OWASP A01 · ASVS V2.1 · SOC2 CC6.1 File: Source/Backend/src/app.ts:1–54

The Express application mounts all routers (workItemsRouter, workflowRouter, dashboardRouter, intakeRouter) with zero authentication or authorization middleware. No auth library (passport, jsonwebtoken, express-jwt) is installed. Every endpoint — state-transition operations (/approve, /reject, /dispatch, /route), dashboard data, and intake webhooks — is publicly accessible to any HTTP client with network access to port 3001. This finding is the root amplifier for PEN-002, PEN-003, PEN-004, PEN-005, PEN-006, PEN-007, and COMP-006.

Remediation Add JWT bearer token validation or session middleware as the first global middleware before all route mounts. For webhooks specifically, add HMAC signature validation (see MERGED-002). Consider express-jwt or passport-jwt. Restrict unauthenticated access to health-check and login endpoints only.
PEN-002 Fast-Track Route Override Bypasses Entire Assessment Pod
High Theoretical
Source: pen-tester OWASP A01 · Business Logic Bypass Files: routes/workflow.ts:57 · services/router.ts:66–88

POST /api/work-items/:id/route accepts overrideRoute from the raw request body with no privilege check. When overrideRoute === "fast-track", the entire assessment classification is skipped and the item transitions directly to Approved status. The normal path is backlog → routing → proposed → reviewing → approved, requiring four-role assessment consensus. The override collapses this to a single unauthenticated call. Additionally, arbitrary string values (e.g., "admin-bypass") are written directly to WorkItem.route, corrupting the domain model.

Achieves red-team objective: "Bypass work item state machine to reach an invalid status."

Remediation (1) Require elevated authorization (e.g., role: admin) to use overrideRoute. (2) Validate overrideRoute against the WorkItemRoute enum before processing. (3) Log all override-route actions to the audit trail.
PEN-003 Manual Approve Endpoint Bypasses Assessment Pod — No Auth
High Theoretical
Source: pen-tester OWASP A01 · Business Logic Bypass File: routes/workflow.ts:94–142

POST /api/work-items/:id/approve allows any unauthenticated caller to approve a work item in proposed, reviewing, or routing status. No credential or privilege check is enforced. Combined with MERGED-001, any HTTP client can approve any queued item, bypassing the four-role assessment pod. Items approved this way will have zero assessment records, defeating the quality-gate purpose of the system.

Remediation Restrict this endpoint to authenticated users with pod-lead or admin role. Require the reason field for manual overrides and audit-log all manual approvals.
MERGED-002 Intake Webhooks: No HMAC Signature Verification + Unvalidated Enum Fields
High Theoretical PEN-004 + PEN-005 + SAST-003 + COMP-010
Sources: pen-tester · static-analyzer · compliance-auditor CWE-345 · OWASP A08 File: routes/intake.ts:11–55

Two compounding issues: (1) POST /api/intake/zendesk and POST /api/intake/automated perform no HMAC-SHA256 verification of webhook signatures — any attacker can inject work items into the workflow queue impersonating Zendesk or automated monitors. (2) The type and priority fields accepted by these endpoints are not validated against their respective enums. The main POST /api/work-items endpoint validates both fields, but intake routes do not. Arbitrary strings (e.g., "superadmin", "nuclear") are stored in the in-memory model, corrupt dashboard counters, poison Prometheus metric label cardinality, and cause silent failures in assessment pod type-switch logic.

Remediation (1) Add HMAC-SHA256 webhook signature verification using ZENDESK_WEBHOOK_SECRET env var, matching X-Zendesk-Webhook-Signature. (2) Validate body.type and body.priority against their enum values, returning 400 for invalid input — consistent with the main work-item endpoint.
MERGED-003 Unbounded Pagination Parameter Enables Full Dataset Enumeration
High Theoretical PEN-006 + SAST-004 + COMP-007
Sources: pen-tester · static-analyzer · compliance-auditor CWE-400 · OWASP A01 Files: routes/workItems.ts:69–70 · routes/dashboard.ts:17–18

The ?limit= parameter is parsed with parseInt() and passed to the store with no upper-bound cap. Sending ?limit=999999999 returns the entire dataset in a single unauthenticated response. Combined with MERGED-001, this achieves the red-team objective: "Enumerate all work items without pagination limit enforcement." Additionally, parseInt('abc', 10) returns NaN; result.slice(NaN, NaN) returns [], causing a silent 200 response with empty data — clients assuming empty means "no results" will miss existing records.

Remediation Clamp limit to [1, 100]: const limit = (!isNaN(raw) && raw > 0 && raw <= 100) ? raw : 20;. Apply consistently across all paginated routes. Also validate page is a positive integer.
PEN-007 Cascade Auto-Dispatch Bypasses Team Validation via Dependency Chain
High Theoretical
Source: pen-tester Business Logic Bypass Files: services/dependency.ts:294–299 · store/workItemStore.ts:67–75

The onItemResolved cascade handler calls store.updateWorkItem() directly with status: InProgress and assignedTeam: team without validating the transition against VALID_STATUS_TRANSITIONS. An item can be cascade-dispatched to InProgress from statuses other than Approved if the blocker resolution check is bypassed. Furthermore, computeHasUnresolvedBlockers treats soft-deleted blockers as unresolved (if (!blocker || !RESOLVED_STATUSES.includes(blocker.status))), permanently blocking dependents after a blocker is deleted.

Remediation Add state-machine validation in onItemResolved before cascading dispatch. Treat soft-deleted blockers as "resolved" (not "unresolved") in computeHasUnresolvedBlockers. Add a RESOLVED_STATUSES entry for deleted items or track deletion separately from the status field.
COMP-002 No TLS — All Traffic Transmitted in Cleartext (HTTP-Only)
High Theoretical
Source: compliance-auditor OWASP A02 · ASVS V9 (Communications) · SOC2 CC6.1

Nginx binds HTTP-only on port 80/3001. No TLS termination is configured anywhere in the stack. All API traffic — including any future credential payloads, session tokens, and work item data — is transmitted in cleartext over the network. This fails OWASP-ASVS V9 (Communications Security) and SOC2 CC6.1 (logical access security). Any network observer can read or modify API traffic.

Remediation Configure TLS termination in Nginx with a valid certificate (Let's Encrypt or internal CA). Redirect all HTTP traffic to HTTPS. Add HSTS via Strict-Transport-Security: max-age=31536000; includeSubDomains once TLS is in place.
COMP-006 3 of 4 Required Audit Events Missing from Audit Log
High Theoretical
Source: compliance-auditor SOC2 CC7.1 · SOC2 CC8.1 · ASVS V7 (Logging)

The security.config.yml mandates audit logging for four event types: login_attempt, permission_denied, state_transition, data_export. Only state_transition events are logged (via change history on work items). The other three are absent: there is no authentication system to emit login_attempt, no authorization system to emit permission_denied, and no data export tracking. SOC2 CC7.1 requires detection of security events; without these logs, unauthorized access is undetectable after the fact.

Remediation After implementing authentication (MERGED-001 fix): emit structured audit events for all four types to the logging system. Use a dedicated AuditLogger service separate from the application logger. Ensure audit logs are immutable and separately stored.
COMP-009 No GDPR-Compliant Hard-Delete Capability (Right to Erasure)
High Theoretical
Source: compliance-auditor GDPR Art. 17 · SOC2 CC6.3

All deletes are soft-deletes: items are flagged with deleted: true and remain in the in-memory store indefinitely. The application provides no mechanism to permanently erase personal data. GDPR Article 17 (Right to Erasure) requires that personal data be permanently deleted upon request. With only soft-delete, user data (titles, descriptions, change histories) persists until server restart. Note: this has a secondary operational impact — soft-deleted blockers permanently trap dependent items (PEN-007).

Remediation Implement a DELETE /api/work-items/:id?hard=true endpoint (or equivalent) that permanently removes the record and all associated assessments, change history, and dependency links. Gate hard-delete behind an elevated admin role with audit logging.
SAST-002 Raw Exception Messages Returned to API Clients
Medium Theoretical
Source: static-analyzer CWE-209 · OWASP A05 File: routes/workflow.ts ×7 catch blocks (lines 60, 87, 138, 205, 292, 331, 368)

All seven catch blocks in workflow routes extract err.message and return it verbatim in HTTP 500 responses. While current messages are benign, any future lower-level exception (ORM, file I/O, network) could expose internal paths, data structures, or configuration. The global errorHandler middleware correctly returns a generic message but is never reached because these handlers short-circuit it.

Remediation Log err.message internally (already done) but return only res.status(500).json({ error: 'Internal server error' }) to clients.
MERGED-004 Missing HTTP Security Headers (No helmet Middleware)
Medium Theoretical SAST-005 + COMP-003
Sources: static-analyzer · compliance-auditor CWE-16 · OWASP A05 File: Source/Backend/src/app.ts

No helmet or equivalent middleware is registered. Missing headers: Content-Security-Policy, X-Frame-Options, X-Content-Type-Options: nosniff, Strict-Transport-Security, Referrer-Policy. MIME sniffing is enabled, and there is no clickjacking protection.

Remediation Add import helmet from 'helmet'; app.use(helmet()); as the first middleware. Configure CSP for the frontend origin. Add helmet to Backend/package.json.
MERGED-005 No Rate Limiting on Any Endpoint
Medium Theoretical SAST-006 + COMP-005
Sources: static-analyzer · compliance-auditor CWE-770 · OWASP A04 File: Source/Backend/src/app.ts

No express-rate-limit or equivalent middleware is installed. Vectors: bulk work-item creation via intake endpoints (memory exhaustion), repeated state-transition requests (CPU load), sequential ID enumeration via GET /api/work-items/:id. Compound risk with MERGED-003 (unbounded pagination).

Remediation Add express-rate-limit globally (e.g., 100 req/min) and tighten on intake/write-heavy endpoints (e.g., 10 req/min on /intake/*).
COMP-004 No CORS Policy Configured
Medium Theoretical
Source: compliance-auditor OWASP A05

No CORS middleware (cors package or manual Access-Control-Allow-Origin headers) is registered. Express defaults allow any origin in some configurations. Any web page can make cross-origin requests to the API from a victim's browser, reading unauthenticated responses.

Remediation Add cors({ origin: process.env.ALLOWED_ORIGIN }) middleware and explicitly whitelist the frontend origin.
COMP-008 In-Memory Store Only — No Persistence, No Encryption at Rest
Medium Theoretical
Source: compliance-auditor OWASP A02 · SOC2 CC6.1 · ASVS V8

All data is stored in a Map in the Node.js process heap. A process crash loses all data. There is no persistence layer and no encryption at rest. Any process-level memory dump or core dump exposes all work items in plaintext. SOC2 CC6.1 requires data protection controls for stored sensitive information.

Remediation Migrate to a persistent store (PostgreSQL, SQLite at minimum). Enable encryption at rest at the database or volume level. Add a migration strategy for the current in-memory state.
PEN-008 Prometheus Metrics Endpoint Unauthenticated — Operational Intelligence Leak
Medium Theoretical
Source: pen-tester OWASP A01 · Information Disclosure File: Source/Backend/src/app.ts:34–37

GET /metrics exposes Prometheus counters revealing item creation rates by source/type, routing distribution (fast-track vs full-review), assessment rejection rates, team workload distribution, and Node.js heap/GC/event-loop metrics — all without authentication. This enables attackers to time operations, infer workflow patterns, and enumerate valid label values for further targeting.

Remediation Gate /metrics behind IP allowlist, bearer token, or network-level access control. Consider serving metrics on a separate non-public port.
PEN-009 No Input Length Validation — Memory Exhaustion DoS
Medium Theoretical
Source: pen-tester CWE-400 · OWASP A04 Files: routes/workItems.ts:24–25 · routes/intake.ts:14–15

Neither POST /api/work-items nor the intake endpoints validate title or description length. Express's default 100KB body limit means each item can have a ~99KB description. These are stored in the in-memory Map and serialized into every findAll() result. Bulk insertion of large items causes unbounded memory growth and eventual OOM. No rate limiting is installed to slow such an attack.

Remediation Add maxLength validation: title ≤ 500 chars, description ≤ 10,000 chars. Return 400 for oversized inputs. Also add rate limiting (MERGED-005).
PEN-010 NaN Injection via Pagination Parameters — Silent Data Truncation
Medium Theoretical
Source: pen-tester CWE-682 · Input Validation Files: routes/workItems.ts:68–71 · routes/dashboard.ts:17–18

parseInt('abc', 10) returns NaN. result.slice(NaN, NaN) returns []. The API returns HTTP 200 with {data: [], total: N, page: NaN, limit: NaN} — a client trusting empty data to mean no results will miss all existing records. totalPages becomes Infinity when both are NaN.

Remediation Validate page and limit are integers before use (see MERGED-003). Return 400 for non-numeric values rather than silently defaulting to empty results.
PEN-011 Assessment Records Accumulate Unboundedly on Repeated Assess Calls
Medium Theoretical
Source: pen-tester CWE-400 File: services/assessment.ts:207–210

Each POST /api/work-items/:id/assess call appends four new assessment records to the item's assessments array via [...item.assessments, ...result.assessments]. Cycling an item through the state machine repeatedly (reject → backlog → route → propose → assess) accumulates thousands of records. All are serialized in every GET /api/work-items/:id response and getActivity() output.

Remediation Cap the number of stored assessment records per item (e.g., keep the last 3 assess cycles). Alternatively, link assessments to assessment runs rather than appending inline.
SAST-007 Debug Portal iframe Missing sandbox Attribute
Low Theoretical
Source: static-analyzer CWE-668 · Least Privilege File: Source/Frontend/src/pages/DebugPortalPage.tsx:9–14

The debug portal iframe embeds VITE_PORTAL_URL (defaulting to http://localhost:4200) without a sandbox attribute. Without sandboxing, the embedded page can navigate the top-level frame and execute scripts with full same-origin privileges if co-hosted.

Remediation Add sandbox="allow-scripts allow-same-origin allow-forms" to the iframe element.
PEN-012 Soft-Deleted Blocker Permanently Traps Dependent Work Items
Low Theoretical
Source: pen-tester Business Logic · Data Integrity File: services/dependency.ts:172–210

computeHasUnresolvedBlockers calls store.findById(link.blockerItemId), which returns undefined for soft-deleted items. The code treats undefined as "unresolved": if (!blocker || !RESOLVED_STATUSES.includes(blocker.status)) return true. Deleting a blocker item permanently prevents all dependents from being dispatched. Partially achieves red-team objective: "Access or modify a soft-deleted work item via direct ID reference."

Remediation Treat soft-deleted blockers as resolved in computeHasUnresolvedBlockers. Or store deletion as a terminal status (e.g., status: "cancelled") so the existing resolved-status check handles it.
PEN-013 Missing /complete and /fail Endpoints — State Machine Incomplete
Low Theoretical
Source: pen-tester Business Logic Gap File: routes/workflow.ts (absent endpoints)

VALID_STATUS_TRANSITIONS defines that InProgress → Completed | Failed, but no API endpoints exist for these transitions. Once dispatched to InProgress, a work item cannot advance further. Cascade auto-dispatch logic (onItemResolved) fires on Completed or Rejected — but Completed is unreachable, so dependent items are never cascade-dispatched. The state machine is functionally incomplete.

Remediation Implement POST /api/work-items/:id/complete and POST /api/work-items/:id/fail route handlers. Enforce valid source status (InProgress) before applying the transition.
