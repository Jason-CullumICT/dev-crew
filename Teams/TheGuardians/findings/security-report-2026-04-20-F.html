Top 3 Risks

  1. Zero authentication on every API endpoint — any anonymous caller has full read-write control over the entire application. This is the systemic root cause of all critical findings.
  2. Status field writable via PATCH — the state machine is bypassed entirely by directly setting {"status":"completed"}; confirmed live by the red-teamer in 4 unauthenticated HTTP requests.
  3. Unvalidated actions_url in team dispatches — attacker-controlled URLs (including AWS metadata endpoint 169.254.169.254) are stored and will be fetched when dispatches are processed, creating a live SSRF vector.

Scope Note (RED-007)

Coverage Gap: The pen-tester performed static analysis on Source/Backend/ (workflow engine, routes: /api/work-items) but the test environment (docker-compose.test.yml) runs portal/Backend/ (routes: /api/feature-requests, /api/bugs, /api/cycles). Red-team exploitation was performed against the actual running service. Both services share identical vulnerability classes; the portal service has an additional SSRF vector via actions_url. A full audit of Source/Backend/ against a live instance remains outstanding.

2. Consolidated Findings

Critical — Confirmed (Live Exploit)
F-001 Critical Confirmed (Live Exploit) Authentication Completely Absent — Anonymous Full Control
Merged: SAST-001 · PEN-001 · PEN-002 · COMP-001 | Confirmed by: RED-001, RED-002, RED-003

Description: The Express application (both Source/Backend/ and portal/Backend/) mounts every API router with zero authentication middleware. There is no JWT validator, API key check, session cookie, or any identity mechanism. The red-teamer confirmed that creating resources, mutating state, triggering multi-agent pipelines, and reading all data are all possible without any credentials.

CWE: CWE-306 (Missing Authentication for Critical Function), CWE-862 (Missing Authorization)

Remediation (P1 — blocker) Install and wire authentication middleware (e.g., passport-jwt / jsonwebtoken) in app.ts before all route registrations. Define at minimum two roles: operator (read + workflow actions) and admin (approve/reject/dispatch). Return 401 for missing credentials, 403 for insufficient permissions. This finding unblocks COMP-005, COMP-010, COMP-011 remediation.
F-002 Critical Confirmed (Live Exploit) State Machine Bypass via Unauthenticated PATCH — Full Lifecycle Hijack
Merged: PEN-003 analog · PEN-006 analog | Confirmed by: RED-002, RED-003 — Security config objective ACHIEVED

Description: The status field is directly writable via PATCH /api/feature-requests/:id and PATCH /api/bugs/:id. The red-teamer moved a feature request from potential → voting → approved → in_development → completed in 4 unauthenticated PATCH requests, and a critical bug from reported → triaged → in_development → resolved → closed in 4 requests — bypassing all voting, assessment, and review gates. Security config objective "Bypass work item state machine to reach an invalid status" is confirmed achieved.

CWE: CWE-284 (Improper Access Control), CWE-693 (Protection Mechanism Failure)

Remediation (P1 — blocker) Remove status from the PATCH-updatable fields entirely. Status transitions must only be possible through dedicated workflow endpoints that enforce valid transition rules and require authenticated identity with the appropriate role. Apply this fix to both portal/Backend/ and Source/Backend/.
F-003 Critical Confirmed (Live Exploit) Anonymous AI Voting Pipeline Takeover
Merged: PEN-001 analog · PEN-004 analog | Confirmed by: RED-001

Description: Any anonymous caller can create a feature request and immediately trigger the 5-agent AI voting pipeline (TechFeasibilityAgent, BusinessValueAgent, SecurityReviewAgent, UserImpactAgent, ResourceCostAgent) by POSTing to /api/feature-requests/:id/vote. All 5 agents returned approve votes for a red-team-created item; the status transitioned from potential to voting with 5 fabricated AI votes recorded. The voting trigger endpoint accepts any vote value (including INVALID_VERDICT_XYZ) and fires the pipeline unconditionally.

CWE: CWE-306 (Missing Authentication), CWE-20 (Improper Input Validation)

Remediation (P1 — blocker) Require authentication on all /api/feature-requests mutation routes. The /vote endpoint must validate the vote field against allowed enum values and return HTTP 400 for invalid input. The AI pipeline must not fire on invalid inputs. Apply rate limiting to prevent pipeline flooding (see MED-006).
High — 1 Confirmed (Partial) · 7 Theoretical
F-004 High Confirmed (Partial) SSRF via Unvalidated actions_url in Team Dispatches
Confirmed by: RED-005 (URL stored; callback not triggered in test window)

Description: The POST /api/team-dispatches endpoint accepts an actions_url field with no validation. The red-teamer stored http://169.254.169.254/latest/meta-data/ (AWS instance metadata) and http://localhost:3001/api/bugs (internal self-reference). When the dispatch system processes these records (GitHub Actions callback or internal webhook trigger), it will make HTTP requests to attacker-controlled URLs — enabling SSRF, internal network probing, cloud credential exfiltration, or internal API calls as the server identity.

CWE: CWE-918 (Server-Side Request Forgery)

Remediation (P1) Validate actions_url against an allowlist of permitted GitHub Actions URL patterns (e.g., https://api.github.com/*). Reject any non-HTTPS URLs or RFC-1918/localhost addresses. Add authentication to the team-dispatches endpoint (blocked by F-001).
F-005 High Theoretical IDOR — Direct Object References Without Ownership Checks
Source: PEN-005

Description: Every /:id API endpoint retrieves objects by raw UUID with no ownership or team membership check. Any caller can read, modify, approve, reject, dispatch, or delete any item belonging to any user or team. UUIDs are exposed in list endpoints enabling trivial enumeration. All IDOR findings are directly enabled by F-001 but remain independently exploitable once authentication is added without also implementing authorization checks.

Remediation (P2) After implementing authentication (F-001), add ownership/role checks to all /:id handlers: verify the requesting user's role permits the action, and that the item belongs to the caller's workspace/team scope.
F-006 High Theoretical Unauthenticated Webhook Injection — Zendesk & Automated Intake
Merged: SAST-002 · PEN-004 · COMP-007

Description: POST /api/intake/zendesk and POST /api/intake/automated accept arbitrary JSON payloads with no HMAC-SHA256 signature verification, no shared-secret check, and no IP allowlist. Any external caller can inject arbitrary work items impersonating Zendesk or internal automation. The type and priority fields are not validated against their enums — invalid values silently fall back to defaults.

Remediation (P2) (1) Verify X-Zendesk-Webhook-Signature header via HMAC-SHA256 using process.env.ZENDESK_WEBHOOK_SECRET. Use timingSafeEqual. (2) Require Authorization: Bearer <token> on automated intake, validated against process.env.AUTOMATION_WEBHOOK_TOKEN. (3) Validate type and priority against their enum values; return 400 on invalid input.
F-007 High Theoretical Unauthenticated Manual Assessment / Approval Override
Source: PEN-006 (Source/Backend)

Description: POST /api/work-items/:id/approve is the manual override endpoint for privileged reviewers. It accepts any item in Routing, Proposed, or Reviewing status and moves it directly to Approved, bypassing the full assessment pod. There is no role check, approver identity, or minimum time-in-review guard. This path was exercised theoretically; the analog on the portal service was confirmed via RED-002.

Remediation (P2) Gate the approve endpoint behind an admin role (post F-001). Record the authenticated user's identity in changeHistory. Add an audit log event for manual overrides.
F-008 High Theoretical Cascade Auto-Dispatch Hijack via Unauthenticated Rejection
Source: PEN-008

Description: Rejecting a work item triggers onItemResolved(), which auto-dispatches all dependent items that are Approved and now fully unblocked. An attacker can reject any Proposed/Reviewing item to force unreviewed cascade dispatches of multiple teams' work items — without any manual oversight of the dispatched items.

Remediation (P2) Require authentication + reviewer role on the reject endpoint. Log cascade dispatch events with the triggering actor's identity. Consider adding a human-confirmation step before auto-dispatching items on cascade.
F-009 High Theoretical Dependency Gate Bypass via PATCH blockedBy: []
Source: PEN-009

Description: PATCH /api/work-items/:id accepts a blockedBy: [] field that bulk-replaces all dependency links with no status guard. An attacker can clear all blockers from any Approved item, removing the dispatch gate and enabling immediate dispatch of items that should not proceed until their dependencies resolve.

Remediation (P2) Remove blockedBy from the PATCH-updatable fields. Dependency modifications must only be possible through the dedicated /dependencies endpoint with proper authentication and audit logging.
F-010 High Theoretical No TLS / HTTPS Enforcement
Source: COMP-008 · OWASP ASVS V9.1.1 · SOC2 CC6.1 · GDPR Art. 32

Description: The server listens on plain HTTP. All data in transit — including future authentication tokens once F-001 is resolved — will be transmitted in the clear. The Docker Compose test environment exposes port 3001 directly without a TLS-terminating proxy.

Remediation (P1) Place the application behind a TLS-terminating reverse proxy (nginx or Caddy) with a valid certificate. Add HTTP→HTTPS redirect at the proxy layer. Update Docker Compose to include a TLS proxy service. Once helmet is added (see MED-003), HSTS will be set automatically.
F-011 High Theoretical All Required Audit-Log Events Missing
Merged: COMP-005 · OWASP ASVS V7.2.1 · SOC2 CC7.1

Description: Four events mandated by security.config.yml are absent: login_attempt, permission_denied, state_transition, and data_export. The changeHistory array is a domain record, not a security audit log — it uses hardcoded agent strings rather than authenticated user identity and is mutable alongside the item record rather than append-only in a separate stream.

Remediation (P3 — depends on F-001) After authentication is implemented: emit structured audit events for all four required event types through a dedicated append-only audit logger. Include user identity, timestamp, IP address, and outcome on every event.
Medium — 2 Confirmed · 10 Theoretical
F-012 Medium Confirmed (Live Exploit) Unbounded Pagination / Full Data Enumeration in One Request
Merged: SAST-007 · PEN-010 · COMP-006 | Confirmed by: RED-004 — Security config objective ACHIEVED

Description: GET /api/bugs?limit=999999999 returned all 8 records in a single response with no server-side cap. Security config objective "Enumerate all work items without pagination limit enforcement" confirmed achieved. Additionally, non-numeric ?limit=abc causes parseInt to return NaN, producing undefined pagination metadata (totalPages: NaN).

Remediation (P2 — 1 line fix) Cap limit server-side: const limit = Math.min(Math.max(1, parseInt(req.query.limit as string, 10) || 20), 100); Apply in all list endpoints. Return 400 for non-numeric inputs.
F-013 Medium Confirmed (Partial) Malformed Vote Payload Accepted — AI Pipeline Fires Unconditionally
Source: RED-006 (partial — vote value not validated)

Description: POST /api/feature-requests/:id/vote with {"vote":"INVALID_VERDICT_XYZ"} returned HTTP 200 and triggered the full 5-agent voting pipeline. The vote input field provides no meaningful gate — any value fires the pipeline. Security config objective "Submit a malformed assessment verdict" is partially achieved.

Remediation (P2) Validate vote field against allowed enum values before processing. Return HTTP 400 for unrecognized values. The pipeline must not fire silently on invalid input. Also requires auth (F-001) to be meaningful.
F-014 Medium Theoretical No HTTP Security Headers (Missing Helmet)
Merged: SAST-004 · COMP-002 · OWASP ASVS V14.4.x · ISO 27001 A.14.1.2

Description: helmet is absent from package.json and not applied. Missing: Content-Security-Policy, X-Frame-Options, X-Content-Type-Options, Strict-Transport-Security, Referrer-Policy, Permissions-Policy. Absence enables clickjacking, MIME-sniffing, and cross-origin information leakage.

Remediation (P1 — trivial) npm install helmet then app.use(helmet()) as the first middleware in app.ts. Tune CSP for API-only use case (defaultSrc: ["'none'"]).
F-015 Medium Theoretical No CORS Policy Configured
Merged: SAST-005 · PEN-012 · COMP-003 · OWASP ASVS V14.5.3

Description: No CORS middleware exists. Any web page on any origin can issue cross-origin fetch requests to the API (no cookies or tokens to forge — the API is open). A permissive reverse proxy would further amplify this to full cross-origin data exfiltration.

Remediation (P1 — trivial) npm install cors then app.use(cors({ origin: process.env.ALLOWED_ORIGINS?.split(',') ?? ['http://localhost:5173'], credentials: true })). Never use wildcard origin.
F-016 Medium Theoretical Prometheus Metrics Endpoint Unauthenticated
Merged: SAST-006 · PEN-011

Description: GET /metrics is publicly accessible. It exposes operational counters, domain throughput (items per source/type/team), dispatch rates, and cycle detection frequency. Confirmed by red-teamer: process memory, CPU, heap, FD count, and cycle_feedback_total are all readable without credentials.

Remediation (P2) Restrict /metrics to internal network access via reverse proxy IP allowlist, or require Authorization: Bearer <scrape-token> validated against an env variable.
F-017 Medium Theoretical No Rate Limiting on Any Endpoint
Merged: SAST-009 · COMP-004 · OWASP ASVS V13.2.5

Description: No rate-limiting middleware exists. The intake endpoints and all workflow mutation endpoints accept unlimited requests, enabling resource exhaustion and amplifying the impact of missing authentication. The assessment endpoint runs a synchronous 4-role pod assessment per call — unlimited concurrent calls cause CPU saturation.

Remediation (P2) npm install express-rate-limit. Apply global limiter (100 req/15 min) and strict per-route limiters (10 req/min) on state-transition and voting endpoints.
F-018 Medium Theoretical Stale Dependency Link Creates Permanent Dispatch Block (DoS)
Source: PEN-013

Description: Soft-deleting a blocker item leaves stale blockedBy links on dependent items. computeHasUnresolvedBlockers() treats a not-found blocker as unresolved, permanently blocking dispatch of any item with a link to a deleted blocker. Exploit: create throwaway → add as dependency → delete throwaway → target item permanently undispatchable.

Remediation (P2) In computeHasUnresolvedBlockers(), treat a not-found blocker as resolved (or log a warning and skip it). When soft-deleting an item, automatically clean up or invalidate all outgoing blockedBy links in dependent items. Consider switching to hard delete for blockers (also see COMP-009).
F-019 Medium Theoretical No GDPR Right to Erasure (Hard Delete Missing)
Source: COMP-009 · GDPR Art. 17

Description: DELETE performs only a soft delete. Work item title and description fields may contain PII. GDPR Art. 17 requires the ability to completely erase personal data on request.

Remediation (P3) Implement a purge/hard-delete mechanism (DELETE /api/work-items/:id/purge), admin-only. Alternatively, implement anonymization of PII fields before soft delete.
F-020 Medium Theoretical No Session Management or Timeout Policy
Source: COMP-010 · OWASP ASVS V3.2.1 V3.3.x · SOC2 CC6.2

No session management policy is defined. OWASP ASVS L2 requires ≥128-bit session token entropy, session invalidation on logout, and idle timeout ≤30 minutes. Depends on F-001.

Remediation (P4 — depends on F-001) Use stateless JWT with short-lived access tokens (15 min) and refresh token rotation, or express-session with Redis and 30-minute idle timeout. Implement /auth/logout to invalidate refresh tokens.
F-021 Medium Theoretical No Actor Identity in State-Transition Change History
Source: COMP-011 · SOC2 CC8.1 · OWASP ASVS V7.2.6

All change history entries use hardcoded strings ('user', 'manual-override', 'system') instead of authenticated user identity. SOC2 CC8.1 requires changes to be attributable to specific individuals. Depends on F-001.

Remediation (P3 — depends on F-001) Thread authenticated user identity through to trackUpdates() and buildChangeEntry() calls. Replace literal 'user' strings with the requesting user's ID from the JWT/session.
F-022 Medium Theoretical Debug Portal iframe Missing sandbox Attribute
Merged: SAST-008 · PEN-016 · CWE-1021

DebugPortalPage.tsx embeds an iframe with no sandbox attribute, granting the framed content full JS execution and form submission. The src is sourced from VITE_PORTAL_URL — if this build variable is compromised, an attacker-controlled URL loads inside the application chrome.

Remediation (P3) Add sandbox="allow-scripts allow-same-origin allow-forms" to the iframe. Validate VITE_PORTAL_URL against a trusted origin allowlist at build time.
F-023 Medium Theoretical Internal Error Messages Leaked to API Clients
Merged: SAST-003 · PEN-015 · CWE-209

Every catch block in workflow.ts (6 locations) returns the raw err.message in HTTP 500 responses. Dependency endpoints also reflect internal IDs in 404 error messages, enabling UUID oracle attacks.

Remediation (P2) Return a generic "Internal server error" for all 500-class errors. Log full detail server-side (already done). For 404s, return a generic "Not found" without exposing the ID in the message body.
Low — Theoretical
F-024 Low Theoretical Invalid overrideRoute Corrupts Domain Data
Source: PEN-014

classifyRoute() accepts any truthy string as overrideRoute, storing attacker-controlled strings in the route field. Data corruption only — no status bypass (item transitions to Proposed, not Approved).

Remediation (P3) Validate overrideRoute against Object.values(WorkItemRoute). Return 400 for unrecognized values.
F-025 Low Theoretical Error Handler Logs Full Stack Traces in All Environments
Source: COMP-012 · OWASP ASVS V7.4.1

errorHandler.ts logs err.stack unconditionally, revealing internal module paths and library versions to anyone with log access. Clients correctly receive only a generic error, but log access controls must be verified at deployment.

Remediation (P4) Omit stack traces in production: stack: process.env.NODE_ENV !== 'production' ? err.stack : undefined
