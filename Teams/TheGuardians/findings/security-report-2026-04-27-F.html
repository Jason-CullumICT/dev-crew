Critical — 3 findings (all confirmed)
C-001 Zero Authentication — All API Endpoints Fully Exposed Critical Confirmed (Live Exploit)
Sources: SAST-001 · PEN-001 · COMP-001 · RED-001 | CWE-306, CWE-862
The entire Express application has zero authentication middleware. No JWT verification, session checks, API key validation, or identity enforcement exists anywhere in the middleware chain or route handlers. The red-teamer confirmed live exploitation: created items, modified state, and read all data with a cold unauthenticated curl from an isolated test environment. All 9 API routers and /metrics are fully exposed with HTTP 200 responses to anonymous callers. The CLAUDE.md references admin@example.com / admin123 as login credentials, but no login endpoint or auth gate protects any route.
Remediation: Add authentication middleware as the first Express middleware (e.g., JWT via jsonwebtoken, or API key check): app.use(authMiddleware) before all route registrations. Role-based authorization (RBAC) must additionally gate all state-mutating endpoints (approve, reject, dispatch). This is the single highest-leverage fix — it is a prerequisite for meaningful audit logging, SOC2 CC6.x compliance, and every other finding's severity reduction.
C-002 State Machine Bypass via Force-Approve — Assessment Pod Fully Bypassed Critical Confirmed — Critical Objective ACHIEVED
Sources: PEN-002 · PEN-003 · RED-002 | CWE-284, CWE-862
The red-teamer executed a 3-step exploit chain — all unauthenticated — that bypassed the entire four-role AI assessment pod and forced a work item to approved status with 0 votes cast:
  1. POST /api/feature-requests → item created in potential state, 0 votes
  2. PATCH /api/feature-requests/FR-0003 {"status":"voting"} → transitions directly, skipping AI vote simulation
  3. POST /api/feature-requests/FR-0003/force-approve → status set to approved, votes array empty
Confirmed: item FR-0003 persisted as approved with an empty votes[] array. The pen-tester separately identified that the fast-track override and manual-approve override endpoints (PEN-002, PEN-003) both accept direct state transitions without any authentication or prior assessment gate. This directly achieves pentest objective: "Bypass work item state machine to reach an invalid status."
Remediation: (1) Implement authentication (C-001) — without auth, no state guard can be enforced. (2) Remove or strictly gate the force-approve endpoint behind a privileged role (e.g., admin-only, requiring completed assessment). (3) Add server-side state machine validation that checks prerequisites (votes cast, assessment completed) before any approval transition, regardless of HTTP method or endpoint used.
C-003 Unauthenticated Orchestrator Proxy — SSRF & Internal API Exposure Critical Confirmed (Live Exploit)
Sources: RED-008 (new finding — not in pen-tester map) | CWE-918, CWE-200
The /api/orchestrator/* route proxies all requests — including unauthenticated ones — directly to an internal orchestrator service at http://localhost:8080. Any unauthenticated caller can interact with internal orchestrator APIs by routing requests through this proxy. Additionally, error responses leak the internal service URL: "Orchestrator unreachable at http://localhost:8080" — disclosing internal network topology. This is an SSRF-class vulnerability: an attacker can probe and interact with internal services that should not be reachable from the public API. Note: This finding was discovered by the red-teamer against portal/Backend/; the pen-tester analyzed Source/Backend/ — verify the proxy route exists in both backends.
Remediation: (1) Add authentication + authorization check before the orchestrator proxy route. (2) Strip the internal URL from error responses — return a generic message. (3) Validate and allowlist the subset of orchestrator API paths this proxy is permitted to forward. (4) Consider whether the proxy should be on an internal-only port rather than the public API.
High — 8 findings (3 confirmed, 5 theoretical)
H-001 Missing HTTP Security Headers + No CORS Policy High Theoretical
Sources: SAST-002 · PEN-012 · COMP-005 · COMP-006 | CWE-693, CWE-116
No security headers are set: no Content-Security-Policy, no X-Frame-Options, no Strict-Transport-Security, no X-Content-Type-Options: nosniff, no Referrer-Policy. No CORS policy is enforced — Express's default permits cross-origin requests from any origin. This enables XSS, clickjacking, MIME-sniffing, and cross-origin data theft attacks.
Remediation: Install helmet as the first middleware: app.use(helmet()). Add cors() with an explicit origin allowlist for the frontend domain only.
H-002 Unauthenticated Prometheus Metrics Endpoint — Operational Intelligence Leak High Confirmed
Sources: SAST-004 · PEN-008 · RED-006 | CWE-200
The red-teamer confirmed GET /metrics returns full Prometheus output with zero authentication, including route-by-route request counts, Node.js heap/CPU/GC metrics, and AI voting invocation counters. This exposes internal business throughput, which teams are active, what types of work are processed, and performance characteristics useful for timing attacks.
Remediation: Restrict /metrics to internal network requests or add a Bearer token check against a Prometheus scraper secret. Ensure the metrics port is not exposed externally in Docker/Kubernetes configs.
H-003 Unbounded Pagination Limit — Full Dataset Enumeration in One Request High Confirmed
Sources: SAST-006 · PEN-005 · RED-005 | CWE-20, CWE-770
The red-teamer confirmed all 22 items returned in a single unauthenticated call to GET /api/feature-requests with no pagination. The ?limit=999999 parameter is silently ignored and the entire dataset is returned. The limit query parameter is parsed without an upper-bound check or NaN guard, making this a full data dump and denial-of-service vector as the dataset grows. This achieves pentest objective: "Enumerate all work items without pagination limit enforcement."
Remediation: Cap limit: const limit = Math.min(parseInt(req.query.limit as string) || 20, 100). Add NaN guard: const page = parseInt(req.query.page as string) || 1.
H-004 Soft-Delete Dependency DoS — Permanent Dispatch Block High Partial Confirm
Sources: PEN-006 · RED-003 | CWE-613, CWE-672
The red-teamer confirmed: create a blocker item, add it as a dependency for a target item, then delete the blocker (HTTP 204). The target item permanently shows as blocked: { ready: false, unresolved: [{ id: "BUG-0002", status: "unknown", title: "Unknown" }] }. Root cause: no ON DELETE CASCADE on the dependencies table foreign key. An attacker can permanently prevent any work item from being dispatched by deleting its dependency after linking it. This partially achieves pentest objective: "Access or modify a soft-deleted work item via direct ID reference."
Remediation: Add ON DELETE CASCADE to the dependencies foreign key, or handle orphaned dependencies in the readiness check by treating deleted blockers as resolved. Add an auth gate so only privileged users can delete items that have dependents.
H-005 Unauthenticated Intake Webhooks — No HMAC Verification + No Enum Validation High Theoretical
Sources: SAST-003 · PEN-004 · COMP-004 | CWE-306, CWE-345, CWE-20
/api/intake/zendesk and /api/intake/automated accept arbitrary POST requests with no webhook authentication. Zendesk signs payloads with an HMAC-SHA256 signature in X-Zendesk-Webhook-Signature — this is not verified. Any caller can inject arbitrary work items. Additionally, type and priority fields are passed directly to createWorkItem without enum validation, enabling state corruption via invalid enum values.
Remediation: (1) Add HMAC-SHA256 signature verification middleware for the Zendesk endpoint. (2) Add enum validation for body.type and body.priority mirroring the validation in the main work items POST handler.
H-006 Zero Authorization / RBAC — No Least-Privilege Enforcement High Theoretical
Sources: COMP-002 | CWE-862, CWE-285
Even if authentication is added (C-001), there is no authorization layer: no roles, no permissions, no least-privilege model. Every authenticated user would have full access to every operation — including approve, reject, dispatch, and delete — across all work items. SOC2 CC6.3 requires least-privilege access control; OWASP-ASVS V4 requires authorization on every resource action.
Remediation: Design and implement a RBAC model with at minimum: read-only, operator, and admin roles. Gate approve/reject/dispatch/delete behind operator or admin role checks. Apply checks at the service layer, not just route middleware, to prevent bypass.
H-007 Required Audit Events Missing — Login / Permission Denied / Data Export Unlogged High Theoretical
Sources: COMP-003 | SOC2 CC7.1, OWASP-ASVS V7
Three of four required audit events specified in security.config.yml are absent: login_attempt, permission_denied, and data_export are never logged. Only state_transition events appear in logs (via the workflow logger). SOC2 CC7.1 requires audit logging of access and security events with actor identity. Without auth (C-001), actor identity cannot be logged at all. The existing structured logging infrastructure in utils/logger.ts is solid — it just needs actor context once auth is in place.
Remediation: After implementing auth (C-001): (1) Log login_attempt with outcome and actor IP. (2) Log permission_denied in the auth middleware when a check fails. (3) Log data_export for any bulk list/export endpoint. Inject actor ID into all existing structured log entries via middleware.
H-008 No Rate Limiting on Any Endpoint High Theoretical
Sources: SAST-007 · PEN-007 · COMP-008 | CWE-770
There is no rate limiting middleware anywhere in the Express application. Endpoints triggering expensive computation — POST /:id/assess (sequential 4-role assessment pod) and POST /:id/dependencies (BFS cycle detection) — are unbounded. Combined with the complete absence of authentication, the intake endpoints are especially exposed to spam and resource exhaustion.
Remediation: Add express-rate-limit: app.use(rateLimit({ windowMs: 15 * 60 * 1000, max: 100 })). Apply stricter per-route limits on assessment and intake endpoints.
Medium — 6 findings (all theoretical)
M-001 Error Message Disclosure in 500 Responses Medium Theoretical
Sources: SAST-005 · PEN-014 | CWE-209
All six workflow action endpoints return the raw err.message string to HTTP clients on 500 errors, bypassing the global errorHandler middleware which correctly returns a generic message. Internal exception details, file paths, or connection strings may be disclosed.
Remediation: In each per-route catch block, return res.status(500).json({ error: 'Internal server error' }). Log the full error server-side but never forward raw exception messages to clients.
M-002 Invalid Enum Values Accepted via Intake — Potential State Corruption Medium Theoretical
Sources: SAST-003 (partial) · PEN-009 | CWE-20
The intake routes (/api/intake/zendesk, /api/intake/automated) pass type and priority from the request body directly to createWorkItem without enum validation. Invalid enum values are stored without error, potentially corrupting state machine logic downstream.
Remediation: Add enum validation mirroring the main work items POST handler. Reject requests with invalid type/priority with HTTP 400 and a descriptive error.
M-003 Stored XSS via Unescaped Text Fields Medium Theoretical
Sources: PEN-010 | CWE-79
Work item text fields (title, description) are stored as-is and returned in API responses without sanitization. If the frontend renders these fields as HTML without escaping, stored XSS is possible. Severity is Medium because exploitability depends on the frontend rendering context. Missing CSP (H-001) significantly increases impact if triggered.
Remediation: Sanitize input server-side using a library like DOMPurify or sanitize-html. Ensure the frontend always renders user content as text nodes, not innerHTML. Implement CSP (H-001) as a defense-in-depth layer.
M-004 Docker Container Runs as Root Medium Theoretical
Sources: SAST-009 | CWE-250
The portal Dockerfile has no USER directive. The Node.js process runs as root inside the container. If an attacker achieves code execution via a dependency vulnerability, they have root access within the container, making container escape more impactful.
Remediation: 
RUN addgroup --gid 1001 appgroup && adduser --uid 1001 --gid 1001 --no-create-home appuser
USER appuser
M-005 CI Workflows Use --dangerously-skip-permissions Medium Theoretical
Sources: SAST-010 | CWE-732
Every CI workflow invokes Claude Code with --dangerously-skip-permissions, bypassing interactive permission prompts for filesystem, shell, and network access. In the same environment, ANTHROPIC_API_KEY and CLAUDE_CREDENTIALS secrets are in scope. A prompt-injected or misbehaving AI agent has no friction barrier before executing arbitrary Bash commands or reading sensitive files.
Remediation: (1) Scope --allowedTools as narrowly as possible per agent role (read-only agents should not have Bash). (2) Ensure API secrets are not readable from within Bash tool execution. (3) Consider sandbox mode for high-risk pipelines.
M-006 Unvalidated page/limit Parameters — NaN/Edge Cases Medium Theoretical
Sources: PEN-011 | CWE-20
parseInt('abc', 10) returns NaN, which propagates to slice logic and can cause unexpected behavior. Negative values for page are not guarded. While the primary risk is data dump (H-003), NaN propagation may also cause 500 errors disclosing stack traces (M-001).
Remediation: const page = Math.max(1, parseInt(req.query.page as string) || 1). const limit = Math.min(Math.max(1, parseInt(req.query.limit as string) || 20), 100).
