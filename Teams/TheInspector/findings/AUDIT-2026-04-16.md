## Findings by Severity

### 🔴 DEP-001: Handlebars JavaScript Injection & Prototype Pollution (CRITICAL)

- **Severity:** P1 (CRITICAL)
- **Category:** CVE / JavaScript Injection
- **Package:** `handlebars` @ 4.7.8 (in node_modules, via transitive dependency)
- **Direct Path:** Backend → ts-jest → handlebars
- **Affected Versions:** ≥4.0.0 ≤4.7.8
- **Vulnerabilities (7 distinct):**
  1. **GHSA-2w6w-674q-4c4q** (CRITICAL CVSS 9.8): JavaScript Injection via AST Type Confusion
     - Attacker supplies malicious template → arbitrary code execution
     - No authentication required, network-accessible
  2. **GHSA-3mfm-83xf-c92r** (HIGH CVSS 8.1): JavaScript Injection via @partial-block tampering
  3. **GHSA-xhpv-hc6g-r9c6** (HIGH CVSS 8.1): JavaScript Injection when object passed as dynamic partial
  4. **GHSA-9cx6-37pm-9jff** (HIGH CVSS 7.5): DoS via malformed decorator syntax
  5. **GHSA-xjpj-3mr7-gcpf** (HIGH CVSS 8.3): JavaScript Injection in CLI precompiler
  6. **GHSA-2qvq-rjwj-gvw9** (MODERATE CVSS 4.7): Prototype Pollution → XSS via partial injection
  7. **GHSA-7rx3-28cr-v5wh** (MODERATE CVSS 4.8): Prototype method access control gap

- **Impact:** If the Backend uses Handlebars for *any* template rendering (logging, response generation, CLI tools), attackers can inject JavaScript that executes during template compilation. Runtime impacts include:
  - Remote code execution (RCE) if templates are user-controlled
  - Information disclosure (environment variables, memory contents)
  - State corruption if templates affect work item serialization
  
- **Current Status:** Backend does NOT appear to use Handlebars directly (not in dependencies), but ts-jest brings it in for **development/test builds only**. This **reduces** runtime risk but does NOT eliminate build-time risks (malicious test fixtures, build artifacts shipped in package).

- **Recommended Fix (Urgent):**
  ```bash
  # Option 1: Update ts-jest to latest (uses handlebars ≥4.7.9)
  cd Source/Backend && npm install ts-jest@latest --save-dev
  
  # Option 2: Audit ts-jest usage—if not needed for tests, remove it
  # and use native Node.js test runner or Jest without TypeScript compilation
  ```
  
- **Fix Status:** ⚠️ **BLOCKED** — ts-jest must update first. Check upstream.
- **Cross-ref:** [ESCALATE → TheGuardians] — JavaScript injection is a security violation; validate that build artifacts don't embed malicious code.

## Supply Chain Risks

### 🟠 Risk 1: Transitive Handlebars via ts-jest
- **Issue:** ts-jest brings handlebars as a test-only dep. Vulnerable to RCE during test execution.
- **Mitigation:** Keep ts-jest updated OR migrate to native TypeScript support in Jest (jest-ts-preset).
- **Action:** Monitor ts-jest releases; update immediately when they patch Handlebars.

### 🟠 Risk 2: Single-Maintainer esbuild
- **Issue:** esbuild (built-in to Vite) is primarily maintained by one person (Evan Wallace).
- **Mitigation:** esbuild is widely used and audited. Keep Vite updated (which pins esbuild versions).
- **Action:** No immediate action, but monitor Evan Wallace's GitHub for security advisories.

### 🟠 Risk 3: Vite/Vitest Rapid Release Cycle
- **Issue:** Vite has fast iteration (major version bumps annually). Staying N-1 major versions means missing security patches.
- **Mitigation:** Plan quarterly Vite+Vitest upgrades.
- **Action:** Schedule Frontend Vite → v8 upgrade in next sprint.

### ✅ Positive: Monorepo Approach
- All source code is co-located, reducing supply chain surface (no external package links to private APIs).
- Lock files are checked in → reproducible builds ✅.
