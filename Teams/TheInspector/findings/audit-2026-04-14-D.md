### QO-001: Traceability Enforcer Targets Plans, Not Specifications
- **Severity:** P1
- **Category:** spec-drift / architecture-violation
- **File:** `tools/traceability-enforcer.py:49-57`
- **Detail:** The enforcer auto-selects the *most recently modified* `requirements.md` under `Plans/`. On this run it targeted `Plans/self-judging-workflow/requirements.md` (13 FRs: FR-WF-001 – FR-WF-013) and reported **TRACEABILITY PASSED**. The 85 requirements in `Specifications/dev-workflow-platform.md` are completely ignored. This creates compliance theater — the gate turns green while the authoritative spec is 82% uncovered.
- **Recommendation:** Either (a) point the enforcer at `Specifications/` directly via `--file Specifications/dev-workflow-platform.md`, or (b) update the Specifications to reflect the current product and map its IDs to the FR-WF-*/FR-dependency-* scheme the implementation uses. The CLAUDE.md architecture rule *"Specs are source of truth — implementation traces to specs, never the other way around"* is inverted here.
- **Cross-ref:** requirements-reviewer should reconcile Specifications vs Plans.

### QO-002: Specifications Describe a Different Product (Mass Spec Drift)
- **Severity:** P1
- **Category:** spec-drift
- **File:** `Specifications/dev-workflow-platform.md` (FR-001 — FR-069)
- **Detail:** `Specifications/dev-workflow-platform.md` defines 69 functional requirements for a **SQLite-backed** web app managing Feature Requests, Bug Reports, Development Cycles, Votes, Learnings, and Features (the "portal" product). The actual implementation in `Source/` is a completely different product: an **in-memory workflow engine** with WorkItems, routing pods, assessment pods, and dispatch gating. Domain entities, data persistence strategy, API surface, and FR IDs are all different. Not a single FR-001 through FR-069 has a `// Verifies:` comment anywhere in `Source/`.
- **Recommendation:** Specs must be updated to describe the current product before any new feature work begins. Create a new spec (e.g., `Specifications/workflow-engine.md`) that maps to FR-WF-001 – FR-WF-013 plus FR-dependency-*. Archive or clearly mark FR-001 – FR-069 as describing a superseded product.
- **Cross-ref:** QO-001 (traceability enforcer is masking this gap).

### QO-003: GET /api/search Route Untethered — Tests Will Fail
- **Severity:** P1
- **Category:** untested / spec-drift
- **File:** `Source/Backend/tests/routes/search.test.ts` (all 5 tests) / `Source/Backend/src/app.ts`
- **Detail:** Five tests in `search.test.ts` cover `GET /api/search?q=` (FR-dependency-search). The test file's own header documents the problem: *"the GET /api/search endpoint is NOT wired into app.ts. These tests will FAIL until the route is implemented."* The route is absent from `app.ts` — there is no `app.use('/api/search', ...)` registration. All 5 tests hit a 404 and fail.
- **Recommendation:** Implement `Source/Backend/src/routes/search.ts` (cross-entity title/description search over the in-memory store) and register it in `app.ts`: `app.use('/api/search', searchRouter)`. The test file already defines the full contract.
- **Cross-ref:** [ESCALATE → TheFixer] implementation gap.

### QO-004: Route Handlers Call the Store Directly (Service Layer Bypass)
- **Severity:** P2
- **Category:** architecture-violation
- **File:** `Source/Backend/src/routes/workItems.ts:44,73,79,134,141` / `Source/Backend/src/routes/workflow.ts:44,119,175,269`
- **Detail:** CLAUDE.md rule: *"No direct DB calls from route handlers — use the service layer."* The following store calls appear directly in route handlers with no service intermediary:
  - `workItems.ts`: `store.createWorkItem()`, `store.findAll()`, `store.findById()`, `store.updateWorkItem()`, `store.softDelete()`
  - `workflow.ts`: `store.findById()`, `store.updateWorkItem()` (bypassed in approve/reject/dispatch handlers — business logic for state transitions runs in-line)
  
  The approve and reject handlers in `workflow.ts` build `changeHistory` entries, push them into the item, and call `store.updateWorkItem()` directly — this is domain business logic that belongs in a service (e.g., `workItemService.approve(id, reason)`).
- **Recommendation:** Extract CRUD and state-transition logic from route handlers into a `workItemService.ts`. Route handlers should be thin: validate input → call service → return response. This also improves testability (unit-test the service, integration-test the route).
- **Cross-ref:** [ESCALATE → TheFixer].

### QO-005: FR-dependency-seed Never Implemented in Source/
- **Severity:** P2
- **Category:** spec-drift / untested
- **File:** `Specifications/dev-workflow-platform.md:475` / `Plans/dependency-linking/requirements.md:46`
- **Detail:** `FR-dependency-seed` requires idempotent seed data (BUG-0010 blocked by 5 bugs; 3 FR blockers). The Plans status tracker explicitly marks it **"❌ Missing"**: *"No seed.ts file exists in portal/Backend/src/database/"*. In `Source/` there is also no seed mechanism. The spec requires this data to be present after a fresh setup and the GET endpoints to return correct `blocked_by` arrays.
- **Recommendation:** Implement a seed function in `Source/Backend/src/` that populates known dependency fixtures on startup (idempotent via ID-check). The traceability comment `// Verifies: FR-dependency-seed` must appear in the implementation.
- **Cross-ref:** [ESCALATE → TheFixer].

### QO-006: `eslint-disable` Without Justification Comments
- **Severity:** P3
- **Category:** pattern-violation
- **File:** `Source/Frontend/src/components/DependencyPicker.tsx:82` / `Source/Frontend/src/hooks/useWorkItems.ts:63`
- **Detail:** Both files suppress `react-hooks/exhaustive-deps` with no inline explanation of *why* the full dependency array would be incorrect. The architecture rule says *"Never swallow errors silently"* — the same principle extends to suppressed lint warnings: they mask real stale-closure bugs if the suppression is wrong.
- **Recommendation:** Add a one-line comment above each suppression explaining the deliberate exclusion (e.g., `// filters object identity would retrigger on every render — individual fields are stable primitives`). This documents intent and prevents future contributors from accidentally widening the exclusion.

### QO-007: Duplicate Test Files for WorkItemDetailPage
- **Severity:** P3
- **Category:** test quality
- **File:** `Source/Frontend/tests/WorkItemDetailPage.test.tsx` AND `Source/Frontend/tests/pages/WorkItemDetailPage.test.tsx`
- **Detail:** Two independent test files cover the same component. The root-level file uses `FR-WF-011` Verifies comments inline. The `pages/` subdirectory file also has `FR-WF-011` comments. They use different mock shapes (slightly different `WorkItem` stubs) and may diverge over time. Duplicate tests create maintenance overhead and can give false confidence when one file is updated but not the other.
- **Recommendation:** Consolidate into a single canonical location (`tests/pages/WorkItemDetailPage.test.tsx`). Remove `tests/WorkItemDetailPage.test.tsx` after verifying all test cases are covered.

### QO-009: Two Logger Abstraction Files in Backend
- **Severity:** P3
- **Category:** architecture-violation
- **File:** `Source/Backend/src/logger.ts` vs `Source/Backend/src/utils/logger.ts`
- **Detail:** `src/logger.ts` is a thin re-export of `src/utils/logger.ts`. Two import paths exist for the same logger. Some files import `from '../logger'`, others import `from '../utils/logger'`. This violates *"Shared types are single source of truth — no inline type re-definitions across layers"* in spirit: the logger should have one canonical import path.
- **Recommendation:** Choose one canonical path (`src/utils/logger.ts`) and update all imports to use it. Delete `src/logger.ts` or make it the sole real implementation with `utils/logger.ts` removed.
