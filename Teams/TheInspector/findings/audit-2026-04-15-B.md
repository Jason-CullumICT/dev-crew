## Findings

### QO-001: GET /api/search route is missing — test suite will fail
- **Severity:** P2
- **Category:** spec-drift / untested
- **File:** `Source/Backend/tests/routes/search.test.ts` (5 test cases) | no matching `Source/Backend/src/routes/search.ts`
- **Detail:** `FR-dependency-search` requires a `GET /api/search?q=` cross-entity typeahead endpoint on the Source/Backend workflow engine. The test file (`search.test.ts`) was written first and explicitly states in its header: *"GET /api/search endpoint is NOT wired into Source/Backend/src/app.ts. These tests will FAIL until the route is implemented."* The route file does not exist and the endpoint is not registered in `app.ts`. The traceability enforcer **passes** (it finds the `// Verifies: FR-dependency-search` comment in the test file) even though the production code is absent. This is a false-pass in the enforcement pipeline.
- **Recommendation:** Implement `Source/Backend/src/routes/search.ts` — search across the in-memory store's non-deleted work items by title/description; return `{data: T[]}`. Register at `/api/search` in `app.ts`. Remove the "will FAIL" header comment once done.
- **Cross-ref:** TheFixer (code gap); traceability enforcement gap surfaces separately in QO-002.

### QO-002: Traceability enforcer blind spot — portal/ never scanned
- **Severity:** P2
- **Category:** architecture-violation / spec-drift
- **File:** `tools/traceability-enforcer.py` (line: `source_dirs = ["Source", "E2E"]`)
- **Detail:** The enforcer hardcodes scan directories as `["Source", "E2E"]`. The `portal/` directory implements the entire dev-workflow-platform spec (FR-001 through FR-069, plus pipeline and traceability extension FRs). This system has 30 test files and 1,068 `// Verifies:` comments, but **zero of them are ever validated** by the CI gate. Any future deletion or drift of these traces would pass the gate silently. Additionally, the `platform/` orchestrator implements the tiered-merge-pipeline spec (FR-TMP-001 through FR-TMP-010) and is also invisible to the enforcer.
- **Recommendation:** Extend the enforcer's `source_dirs` to include `portal/` (and optionally `platform/`). Either update the hardcoded list or honour the `source.dirs` list from `Teams/TheInspector/inspector.config.yml` which already lists `"Source/"` — extend it with `"portal/"`.
- **Cross-ref:** QO-003 (related enforcer defect).

### QO-003: Traceability enforcer produces false-failure on dependency-linking plan
- **Severity:** P2
- **Category:** pattern-violation / tooling
- **File:** `tools/traceability-enforcer.py`, `Plans/dependency-linking/requirements.md`
- **Detail:** Running `python3 tools/traceability-enforcer.py --plan dependency-linking` fails with 7 "missing" requirements: `FR-0002`, `FR-0003`, `FR-0004`, `FR-0005`, `FR-0007`, `FR-070`, `FR-085`. These are **spurious matches** — the enforcer regex `FR-[A-Z0-9-]+` grabs inline references in the requirements prose (seed data examples like "BUG-0010 blocked_by BUG-0003" and spec cross-references like "Specifications/dev-workflow-platform.md (FR-070 — FR-085)"). The actual plan FRs are `FR-dependency-*` IDs, all of which ARE present in source. The tool would mislead an agent into chasing phantom missing requirements.
- **Recommendation:** Tighten the extraction regex to only match IDs that appear in table column position (e.g., require the ID to be at the start of a Markdown table cell: `\|\s*(FR-[A-Z0-9-]+)\s*\|`). Alternatively add an allow-list or a `## Requirements` section header to scope extraction.
- **Cross-ref:** QO-002.

### QO-006: Dual logger abstraction in Source/Backend — DRY violation
- **Severity:** P3
- **Category:** pattern-violation
- **Files:**
  - `Source/Backend/src/utils/logger.ts` — actual structured logger (32 lines)
  - `Source/Backend/src/logger.ts` — compatibility re-export wrapper (34 lines)
- **Detail:** Two logger files exist because different coders imported logger differently: some routes use `import logger from './logger'` (default export) while others use `import { logger } from './utils/logger'` (named export). The wrapper normalizes the interface difference. This violates DRY, adds indirection, and means two import paths exist for what should be a single logger abstraction. The CLAUDE.md rule "Use the project's logger abstraction, never `console.log`" is ambiguous when two valid imports exist.
- **Recommendation:** Consolidate: update all imports to use one path (suggest `./utils/logger`), add a default export to `utils/logger.ts`, and delete `src/logger.ts`.
- **Cross-ref:** None (internal cleanup).

### QO-007: Suppressed ESLint react-hooks/exhaustive-deps in two production files
- **Severity:** P3
- **Category:** pattern-violation
- **Files:**
  - `Source/Frontend/src/components/DependencyPicker.tsx:82`
  - `Source/Frontend/src/hooks/useWorkItems.ts:63`
- **Detail:** Both files use `// eslint-disable-next-line react-hooks/exhaustive-deps` to suppress the exhaustive dependencies rule. This masks stale closure bugs — if the hook's dependencies are incomplete, the component may read outdated values without warning. This also sets a bad precedent for agents who might copy the pattern.
- **Recommendation:** Fix the hook dependency arrays properly. In `DependencyPicker.tsx`, extract the search callback with `useCallback` and include it in the effect deps. In `useWorkItems.ts`, audit what the missing dep is and either include it or use a ref to stabilize it.
- **Cross-ref:** None.

### QO-008: Portal dependency traceability uses non-standard FR-ID naming
- **Severity:** P3
- **Category:** spec-drift
- **Files:** `portal/` — all dependency-linked code
- **Detail:** The dependency-linking plan defines canonical IDs (`FR-dependency-types`, `FR-dependency-service`, `FR-dependency-schema`, etc.). The portal source uses a different naming convention: `FR-dependency-linking`, `FR-dependency-cycle-detection`, `FR-dependency-ready-check`, `FR-dependency-dispatch-gating`. These non-canonical IDs cannot be cross-referenced against the plan's requirement table. If an agent runs `--plan dependency-linking` enforcement, the portal traces will not satisfy the check even after QO-002/QO-003 are fixed.
- **Recommendation:** Align portal Verifies comments to the canonical plan IDs. This is a search-and-replace across portal/. Alternatively, update the plan's requirement IDs to match what was implemented — but keep the plan authoritative.
- **Cross-ref:** QO-002, QO-003.

### QO-010: Traceability enforcer default mode may target stale plan
- **Severity:** P4
- **Category:** tooling / pattern-violation
- **File:** `tools/traceability-enforcer.py` — `get_active_requirements` fallback
- **Detail:** When run without `--plan` or `--file`, the enforcer picks the **most recently modified** `requirements.md` under `Plans/`. As of this audit that is `Plans/self-judging-workflow/requirements.md`. If an agent runs the enforcer to validate dependency-linking work (which is the active plan per the open delta items), the enforcer silently targets the wrong requirements file and reports a false-pass. CLAUDE.md's verification gate section says to run `python3 tools/traceability-enforcer.py` without arguments — meaning the enforcer must be reliable with no args.
- **Recommendation:** Either (a) add a `ACTIVE_PLAN` marker file that the enforcer respects, or (b) require agents to always pass `--plan` in verification gates, and update CLAUDE.md to specify the flag. Document the hazard in the tool's usage message.
- **Cross-ref:** QO-003.
