### QO-001: Traceability Enforcer Scans Only Plans/ — Specifications/ Is Never Validated
- **Severity:** P1
- **Category:** spec-drift / architecture-violation
- **File:** `tools/traceability-enforcer.py:16–57`
- **Detail:** The enforcer targets only the most-recently-modified `requirements.md` inside `Plans/`. The three domain spec files in `Specifications/` — the documents CLAUDE.md calls "the most critical documents" and "Domain truth" — are never validated by the automated gate. Running the enforcer against the default (Plans/self-judging-workflow/requirements.md) returns PASSED, giving a false green signal while 76+ requirements across the other two specs remain unimplemented.
- **Reproduction:** `python3 tools/traceability-enforcer.py` → PASSED (only checks Plans/self-judging-workflow). `python3 tools/traceability-enforcer.py --file Specifications/dev-workflow-platform.md` → 76 MISSING.
- **Recommendation:** Extend the enforcer to accept a `--spec-dir Specifications/` flag, or add a secondary gate step that runs `--file` against each `.md` in `Specifications/`. Update the CLAUDE.md verification-gates block accordingly.
- **Cross-ref:** QO-002, QO-003

### QO-002: Major Spec Drift — dev-workflow-platform.md FR-001 to FR-069 Unimplemented
- **Severity:** P1
- **Category:** spec-drift
- **File:** `Specifications/dev-workflow-platform.md` (entire document)
- **Detail:** `Specifications/dev-workflow-platform.md` defines 69 functional requirements (FR-001 through FR-069) for a SQLite-backed development workflow platform with feature request management (FR-005–012), bug tracking (FR-013), development cycles with pipeline orchestration (FR-014–016, FR-033–049), cycle traceability (FR-050–069), dashboard APIs (FR-017–018), learnings & features APIs (FR-019–020), OpenTelemetry (FR-021), and 9 frontend pages (FR-022–030). **None of these are implemented in `Source/`.** The `Source/` directory implements `Specifications/workflow-engine.md` instead (a different system: an in-memory work-item engine). Enforcer reports 76 MISSING items when pointed at this spec.
- **Recommendation:** Determine canonical mapping: Is `dev-workflow-platform.md` a future spec (backlog) or a spec that `Source/` should already satisfy? If future, annotate it clearly (`## Status: Planned`) and add it to a roadmap. If current, open a P1 implementation gap and run TheATeam pipeline against it. Either way, the automated gate must be extended to catch this.
- **Cross-ref:** QO-001 (enforcer blind spot allows this to go undetected)

### QO-004: Missing Prometheus Histogram — dependencyCheckDuration
- **Severity:** P2
- **Category:** spec-drift
- **File:** `Source/Backend/src/metrics.ts`
- **Detail:** `FR-dependency-metrics` explicitly requires 4 Prometheus metrics: `dependencyOperations` counter, `dispatchGatingEvents` counter, `dependencyCheckDuration` histogram, and `cycleDetectionEvents` counter. The metrics file implements only 3 of the 4 — the `dependencyCheckDuration` histogram is absent. This was listed first in the acceptance criteria comment in inspector.config.yml under `critical_operations`. Without this histogram, there is no latency visibility into the BFS cycle-detection path, which is the most computationally expensive operation in the dependency service.
- **Recommendation:** Add to `metrics.ts`:
  ```ts
  // Verifies: FR-dependency-metrics — dependency_check_duration_ms
  export const dependencyCheckDurationHistogram = new Histogram({
    name: 'dependency_check_duration_ms',
    help: 'Duration of dependency readiness checks in milliseconds',
    labelNames: ['operation'] as const,
    registers: [registry],
    buckets: [1, 5, 10, 25, 50, 100],
  });
  ```
  Then instrument `hasUnresolvedBlockers()` and `isReady()` in `dependency.ts`.
- **Cross-ref:** [ESCALATE → TheFixer]

### QO-005: Silent Error Swallowing in Frontend Catch Blocks — No Logging
- **Severity:** P2
- **Category:** architecture-violation
- **Files:**
  - `Source/Frontend/src/components/DependencyPicker.tsx:97-100`
  - `Source/Frontend/src/pages/WorkItemDetailPage.tsx:34-35`
  - `Source/Frontend/src/pages/CreateWorkItemPage.tsx:50-52`
  - `Source/Frontend/src/hooks/useDashboard.ts:48-52`
  - `Source/Frontend/src/hooks/useWorkItems.ts:53-56, 105-108`
- **Detail:** Architecture rule states: "every `catch` block must either re-throw, log with full context, or explicitly document why the error is intentionally suppressed." Five locations in the frontend set error state (surfacing to the user) but do not log the error. There is no frontend logger abstraction — an oversight in the architecture that makes this rule unenforceable on the frontend. The `DependencyPicker.tsx:56` bare `catch {}` is the exception: it has a clear comment explaining intentional suppression and is acceptable.
- **Root cause:** No `Source/Frontend/src/utils/logger.ts` exists. The backend has `logger.ts` but this was never ported to the frontend.
- **Recommendation:** Create a minimal frontend logger (`src/utils/logger.ts`) that writes structured entries to `console.error` only in development and is a no-op in production (or routes to a remote error sink). Then instrument each of the 5 catch sites.
- **Cross-ref:** [ESCALATE → TheFixer] for the implementation; architecture owners should update the rule to clarify frontend exception.

### QO-006: eslint-disable Suppressing react-hooks/exhaustive-deps — Undocumented
- **Severity:** P3
- **Category:** pattern-violation
- **Files:**
  - `Source/Frontend/src/components/DependencyPicker.tsx:82`
  - `Source/Frontend/src/hooks/useWorkItems.ts:63`
- **Detail:** Two instances of `// eslint-disable-next-line react-hooks/exhaustive-deps` suppress the exhaustive-deps rule without explaining WHY the deps array is intentionally incomplete. The rule exists to catch stale-closure bugs. In `DependencyPicker.tsx:82`, the suppressed deps for `handleAdd` appear to be `[selectedIdSet, blocksIdSet, currentItemDocId]` — these ARE in the deps array, making the suppress potentially unnecessary or indicating a missing dep. In `useWorkItems.ts:63`, the suppressed line is in a `useCallback` whose deps may legitimately omit state setters, but this is not documented.
- **Recommendation:** Add an inline comment at each site explaining the specific reason (e.g., `// deps intentionally omitted: setters are stable React guarantees`). Alternatively, verify whether the suppression is still needed with the current deps array.

### QO-009: Enforcer Extracts Seed Data IDs as Requirement IDs (False Extraction)
- **Severity:** P3
- **Category:** pattern-violation
- **File:** `tools/traceability-enforcer.py:64`, `Specifications/dev-workflow-platform.md` (FR-dependency-seed section)
- **Detail:** The enforcer uses pattern `FR-[A-Z0-9-]+` to extract requirement IDs. In the `FR-dependency-seed` row, the spec text references seeded work item IDs (`FR-0003`, `FR-0004`, `FR-0005`, `FR-0007`) — which are data IDs, not requirement IDs. These get extracted as if they were requirements and then appear as MISSING in enforcer output, adding noise to the failure report. Similarly `FR-XXX` and `FR-XXXX` placeholder text from the spec table header are extracted.
- **Recommendation:** Tighten the extraction pattern to require at least 3 digit characters or use a two-phase check: only treat a string as a requirement ID if it appears in a cell that starts a table row (`| FR-...`). Alternatively, add an allowlist of known-false-IDs to ignore.
