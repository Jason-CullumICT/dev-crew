## QO-001: Canonical Specs Describe an Unbuilt Product
- **Severity:** P1
- **Category:** spec-drift
- **File:** `Specifications/dev-workflow-platform.md` (FR-001 to FR-069)
- **Detail:**  
  `Specifications/dev-workflow-platform.md` defines 69 functional requirements for a product with Feature Requests, Bug Reports, AI Voting, Human Approvals, Development Cycles, SQLite persistence, Learning records, and Feature Browser pages. **None of these exist in `Source/`.** The implemented product is an entirely different domain: a self-judging work-item routing engine with an in-memory store.  
  
  Similarly, `Specifications/tiered-merge-pipeline.md` defines FR-TMP-001 through FR-TMP-008 for a risk-classified merge pipeline with Playwright E2E tests, auto-PR creation, and AI review. This also has zero source implementation traces.

  Running `python3 tools/traceability-enforcer.py` reports **PASSED** because it only targets `Plans/self-judging-workflow/requirements.md` — the canonical `Specifications/` folder is never checked.

- **Recommendation:**  
  Either (a) accept that `Specifications/` represents aspirational/future scope and add a prominent header documenting this, or (b) archive `dev-workflow-platform.md` and `tiered-merge-pipeline.md` as superseded and update them to reflect the actual built system. The spec-as-source-of-truth architecture rule is currently violated at the most fundamental level.
- **Cross-ref:** [ESCALATE → TheFixer or requirements-reviewer for spec reconciliation]

## QO-002: Traceability Enforcer Has a Single-Plan Blind Spot
- **Severity:** P1
- **Category:** spec-drift / pattern-violation
- **File:** `tools/traceability-enforcer.py` (line 59 — `max(req_files, key=os.path.getmtime)`)

Running `python3 tools/traceability-enforcer.py` (the required verification gate from CLAUDE.md) currently targets `Plans/self-judging-workflow/requirements.md` and reports PASSED.

  - `--plan image-upload` → **20 FAILURES** (FR-070–FR-089: 0% implemented)

  The verification gate defined in CLAUDE.md silently passes while 57 requirements across 3 Plans have zero traces. Agents relying on the gate for confidence are making incorrect safety assumptions.

- **Recommendation:**

## QO-003: Route Handlers Bypass Service Layer (Architecture Violation)
- **Severity:** P2
- **Category:** architecture-violation
- **Files:**
  - `Source/Backend/src/routes/workItems.ts:12` — `import * as store from '../store/workItemStore'`
  - `Source/Backend/src/routes/workflow.ts:15` — `import * as store from '../store/workItemStore'`
  - `Source/Backend/src/routes/intake.ts:4` — `import * as store from '../store/workItemStore'`
- **Detail:**  
  CLAUDE.md architecture rule: **"No direct DB calls from route handlers — use the service layer."** Three route handlers import directly from `workItemStore` (the persistence layer). While this is an in-memory store rather than a SQL DB, the rule applies: routes contain business logic that should live in services, making this a testability and separation-of-concerns violation.
  
  `workItems.ts` contains filtering, pagination, validation, and PATCH logic (95+ lines) directly in the router. `workflow.ts` calls `store.getWorkItem()`, `store.updateWorkItem()` directly in handlers for route/assess/approve/reject/dispatch actions.

- **Recommendation:**  
  Extract workItemStore calls from route handlers into service functions (e.g., `workItemService.ts` analogous to `assessment.ts`, `router.ts`, `dependency.ts`). Route handlers should only parse request, call service, and format response.
- **Cross-ref:** [backend-coder in TheFixer or TheATeam pipeline]

- **Category:** untested / pattern-violation
- **Files:**
  - `Source/Frontend/tests/WorkItemDetailPage.test.tsx` (368 lines)
  - `Source/Frontend/tests/pages/WorkItemDetailPage.test.tsx` (393 lines)
  - `Source/Frontend/tests/WorkItemListPage.test.tsx` (exists)
  - `Source/Frontend/tests/pages/WorkItemListPage.test.tsx` (exists)
- **Detail:**  
  Two separate test files cover the same `WorkItemDetailPage` component (and similarly for `WorkItemListPage`). Both files have `// Verifies: FR-WF-011` comments. This creates:
  - Risk of contradictory test expectations that are never caught
  - Maintenance burden when the component changes (two files must be updated)

## QO-006: `react-hooks/exhaustive-deps` Suppressed Without Justification
- **Severity:** P3
- **Category:** pattern-violation
- **Files:**
  - `Source/Frontend/src/components/DependencyPicker.tsx:82`
  - `Source/Frontend/src/hooks/useWorkItems.ts:63`
- **Detail:**  
  Both files use `// eslint-disable-next-line react-hooks/exhaustive-deps` without an explanation comment documenting why the omission is intentional. This suppression can mask stale-closure bugs where hooks read outdated values from a previous render without triggering a re-run.  
  
  CLAUDE.md architecture rule: **"Never swallow errors silently — every catch block must either re-throw, log with full context, or explicitly document why the error is intentionally suppressed."** The spirit of this rule extends to lint suppression: suppressed rules need documented rationale.

- **Recommendation:**  
  Add inline comments explaining why the dependency is deliberately excluded (e.g., `// intentionally omit X — would create infinite loop because Y`). If no valid reason exists, fix the hook to include the dependency.
- **Cross-ref:** [frontend-coder]

## QO-008: Multi-Plan `FR-XXX` ID Space Has No Namespace Authority
- **Severity:** P3
- **Category:** spec-drift / architecture-violation
- **Detail:**  
  The project uses multiple overlapping FR namespace conventions:  
  - `FR-WF-XXX` — self-judging workflow (Plans/self-judging-workflow)
  - `FR-dependency-XXX` — dependency linking (Plans/dependency-linking)
  - `FR-NNN` (numeric) — dev-workflow-platform (Specifications/), orchestrated-dev-cycles, dev-cycle-traceability, image-upload
  - `FR-TMP-XXX` — tiered-merge-pipeline (Specifications/)

  There is no authority document mapping these namespaces to their owning spec/plan. New agents picking up a `// Verifies: FR-050` comment have no way to know which spec document or plan it traces to without exhaustive grep.  
  
  The traceability enforcer's pattern `FR-\\d+` (from inspector.config.yml) matches numeric-only IDs but misses `FR-WF-XXX` and `FR-dependency-XXX` identifiers, creating a second blind spot in the tool.

- **Recommendation:**  
  Add an FR namespace registry (a table in CLAUDE.md or a `Specifications/README.md`) that maps each prefix to its owning document. Update inspector.config.yml `traceability` pattern to cover all namespaces.
