## QO-001: Traceability Enforcer Blind to 87% of Codebase

- **Severity:** P1
- **Category:** spec-drift
- **File:** `tools/traceability-enforcer.py:70`
- **Detail:** The enforcer hardcodes `source_dirs = ["Source", "E2E"]` and only reads the alphabetically-last `Plans/*/requirements.md` (when all have equal mtime — currently `self-judging-workflow`). This means:
  - `portal/` (1,073 `Verifies:` annotations, implementing FR-001 to FR-095+) is **never scanned**
  - `platform/` (FR-TMP-001 to FR-TMP-010 implementations) is **never scanned**
  - 7 of 8 plan requirement files (orchestrated-dev-cycles, dependency-linking, dev-workflow-platform, dev-cycle-traceability, duplicate-deprecated-status, image-upload, orchestrator-cycle-dashboard) are **never enforced**
  - Any developer could silently delete a `// Verifies:` comment in `portal/` and CI gates would not catch it
- **Recommendation:** Update `inspector.config.yml` `source.dirs` to include `portal/` and `platform/`. Update enforcer to iterate all `Plans/*/requirements.md` files. Alternatively create a wrapper script that calls enforcer for each plan.
- **Cross-ref:** TheFixer (tooling fix), TheATeam (if inspector config update is needed)

## QO-002: AssessmentResult Domain Type Defined in Service Layer

- **Severity:** P2
- **Category:** architecture-violation
- **File:** `Source/Backend/src/services/assessment.ts:141`
- **Detail:** `AssessmentResult` (containing `verdict`, `assessments[]`, and `targetStatus`) is exported from the service file rather than `Source/Shared/types/workflow.ts`. The architecture rule is "Shared types are single source of truth — no inline type re-definitions across layers." While `AssessmentRecord` and `AssessmentVerdict` are correctly in `Source/Shared/types/workflow.ts`, `AssessmentResult` (the pod aggregate output) is not.
- **Recommendation:** Move `AssessmentResult` to `Source/Shared/types/workflow.ts` and import it in `assessment.ts`. Check if any other service-local types should be promoted to Shared.
- **Cross-ref:** TheFixer

## QO-003: Duplicate Logger Entry Points

- **Severity:** P2
- **Category:** architecture-violation
- **File:** `Source/Backend/src/logger.ts` + `Source/Backend/src/utils/logger.ts`
- **Detail:** Two logger implementations exist in the same package. `src/utils/logger.ts` is the real logger (custom JSON format). `src/logger.ts` is a compatibility shim that wraps it with a different interface signature. Routes can import either one, leading to inconsistent logging behavior (shim has different `normalize()` logic). The architecture rule is "Use the project's logger abstraction" — with two abstractions, there is no single abstraction.
- **Recommendation:** Remove `src/logger.ts` (the shim) and update all imports to use `src/utils/logger.ts` directly. Adjust the utils logger API to be the canonical form that all routes use.
- **Cross-ref:** TheFixer

## QO-004: Silent JSON Parse Swallow in Frontend API Client

- **Severity:** P2
- **Category:** pattern-violation
- **File:** `Source/Frontend/src/api/client.ts:26`
- **Detail:** `const body = await response.json().catch(() => ({}))` silently swallows JSON parse failures without any logging. If a server returns a non-JSON error body (e.g., an HTML 502 from a load balancer), the catch produces `{}` and the thrown error reads `Request failed: 502` with no trace of what the body contained. Architecture rule: "Never swallow errors silently — every catch block must either re-throw, log with full context, or explicitly document why the error is intentionally suppressed." This catch does none of the three.
- **Recommendation:** Add a comment documenting the intent: `// Intentionally ignores parse errors — non-JSON error bodies fall back to status code message`. Or log in development: `if (import.meta.env.DEV) console.warn('Failed to parse error body', parseErr)`.
- **Cross-ref:** TheFixer

## QO-006: ESLint Suppressions Without Explanation

- **Severity:** P3
- **Category:** pattern-violation
- **File:** `Source/Frontend/src/components/DependencyPicker.tsx:82`, `Source/Frontend/src/hooks/useWorkItems.ts:63`
- **Detail:** Both files suppress `react-hooks/exhaustive-deps` without any comment explaining which dependency is intentionally omitted and why. This rule catches stale closure bugs — suppressing it silently risks introducing bugs that are hard to diagnose. The CLAUDE.md architecture rules do not explicitly allow lint suppressions.
- **Recommendation:** Add a comment on each suppression explaining the intent, e.g.: `// eslint-disable-next-line react-hooks/exhaustive-deps — fetchData intentionally omitted to prevent infinite loop; deps are stable refs`.
- **Cross-ref:** TheFixer
