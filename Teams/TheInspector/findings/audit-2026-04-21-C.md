### QO-001: `GET /api/search` endpoint not registered
- **Severity:** P1
- **Category:** spec-drift / untested (runtime failure)
- **File:** `Source/Backend/src/app.ts` (missing route) + `Source/Backend/tests/routes/search.test.ts:1`
- **Detail:** `FR-dependency-search` requires `GET /api/search?q=` for cross-entity typeahead. The route is fully specced in `Source/Shared/api-contracts.md` and has a complete test suite in `tests/routes/search.test.ts`. The test file header explicitly states: *"the GET /api/search endpoint is NOT wired into Source/Backend/src/app.ts… will FAIL until the route is implemented."* Meanwhile `Source/Frontend/src/api/client.ts:101` calls `workItemsApi.searchItems(q)` which hits `/api/search` — the `DependencyPicker` modal is broken at runtime.
- **Recommendation:** Create `Source/Backend/src/routes/search.ts` that filters `store.getAllItems()` by title/description match on `q`, excludes deleted items, returns `{data: WorkItem[]}`. Register in `app.ts` as `app.use('/api/search', searchRouter)`. All 5 test cases in `search.test.ts` must pass.
- **Cross-ref:** FR-dependency-picker (frontend), FR-dependency-integration (BugDetail/FRDetail views)

### QO-002: `pending_dependencies` status missing — dispatch gating deviates from spec
- **Severity:** P2
- **Category:** spec-drift / architecture-violation
- **File:** `Source/Shared/types/workflow.ts:5` (`WorkItemStatus` enum), `Source/Backend/src/routes/workflow.ts:231`
- **Detail:** `FR-dependency-dispatch-gating` specifies: *"Status transitions to `approved`/`in_development` check all blockers; unresolved blockers → set to `pending_dependencies` instead."* `Source/Shared/api-contracts.md` (State Machine Rules) also documents this. However:
  1. `WorkItemStatus` has no `pending_dependencies` value
  2. The dispatch endpoint (`POST /api/work-items/:id/dispatch`) returns `400 {error: "Cannot dispatch: work item has unresolved blocking dependencies"}` instead of setting the item's status to `pending_dependencies`
  3. No auto-advance from `pending_dependencies` → `approved` when blockers resolve (the cascade dispatcher only auto-dispatches items already in `Approved` status)
- **Recommendation:** Add `PendingDependencies = 'pending_dependencies'` to `WorkItemStatus`. Update `VALID_STATUS_TRANSITIONS` to include transitions from/to it. Change dispatch gating to set status rather than return 400. Update cascade in `onItemResolved` to also advance items in `PendingDependencies` → `Approved` → `InProgress`.
- **Cross-ref:** QO-001 (search must work for DependencyPicker which triggers this flow)

### QO-005: Dual logger modules — inconsistent import paths across codebase
- **Severity:** P2
- **Category:** architecture-violation
- **Files:** `Source/Backend/src/logger.ts`, `Source/Backend/src/utils/logger.ts`
- **Detail:** Two logger implementations exist side-by-side. Routes/services import from `./logger` (compat default export); the store imports from `../utils/logger` (named export). The compat wrapper `logger.ts` was created to normalize `{msg: '...', ...ctx}` object-style calls (used by most routes) vs. `(msg, ctx)` positional calls (used by the store). While no functional bug exists, this is P2 technical debt: adding a new logger call requires knowing which pattern to follow, and a future refactor of the compat wrapper could silently break the store.
- **Recommendation:** Pick one interface and consolidate. Prefer the object-style `{msg, ...ctx}` as it's richer and used by the majority. Migrate the store to use `logger.info({msg: '...', ...fields})` and remove `logger.ts`. Update the canonical source to `utils/logger.ts`.

### QO-006: `LOG_LEVEL` env var not respected — debug logs always emitted
- **Severity:** P3
- **Category:** architecture-violation
- **File:** `Source/Backend/src/utils/logger.ts:17`
- **Detail:** CLAUDE.md architecture rule: "`LOG_LEVEL` env var controls verbosity." The `emit()` function always writes every log level to stdout regardless of `LOG_LEVEL`. Debug logs are noisy in production.
- **Recommendation:** Add level filtering to `emit()`:
  ```ts
  const LOG_LEVELS = ['debug', 'info', 'warn', 'error'];
  const minLevel = LOG_LEVELS.indexOf(process.env.LOG_LEVEL ?? 'info');
  if (LOG_LEVELS.indexOf(level) < minLevel) return;
  ```
