## Findings

### QO-001: Traceability Enforcer Blind Spot — Specifications/ Not Checked
- **Severity:** P2
- **Category:** spec-drift / architecture-violation
- **File:** `tools/traceability-enforcer.py`
- **Detail:** The enforcer scans `Plans/*/requirements.md` for FR IDs (selecting the most recently modified file). It never scans `Specifications/*.md` which is explicitly declared as "The most critical documents" and "domain truth" in CLAUDE.md. Two full specs — `dev-workflow-platform.md` (69 FRs) and `tiered-merge-pipeline.md` (10 FRs) — contain canonical requirements that are completely invisible to the enforcer. Every agent team that runs `python3 tools/traceability-enforcer.py` believes compliance is achieved while 79 spec requirements go unchecked.
- **Recommendation:** Extend the enforcer to also scan `Specifications/` as an optional mode (`--specs`). Add a wrapper that runs both Plans and Specs scans in the verification gate, or add a note in CLAUDE.md clarifying which specs are "future product" vs "current implementation scope" to prevent confusion.
- **Cross-ref:** This structural gap means every agent that trusts the green TRACEABILITY PASSED banner is operating on incomplete data.

### QO-002: Architecture Violation — Route Handlers Import Store Directly
- **Severity:** P2
- **Category:** architecture-violation
- **Files:**
  - `Source/Backend/src/routes/intake.ts:4` — `import * as store from '../store/workItemStore'`
  - `Source/Backend/src/routes/workItems.ts:12` — `import * as store from '../store/workItemStore'`
  - `Source/Backend/src/routes/workflow.ts:15` — `import * as store from '../store/workItemStore'`
- **Detail:** CLAUDE.md architecture rule: "No direct DB calls from route handlers — use the service layer." The in-memory `workItemStore` is the persistence layer (equivalent to a DB). All three route files import and call it directly — `store.findById`, `store.createWorkItem`, `store.updateWorkItem`, `store.softDelete`, `store.findAll`, `store.getAllItems`. Only a subset of operations go through services (router, assessment, changeHistory, dependency). The CRUD pathway in `workItems.ts` entirely bypasses any service layer.
- **Recommendation:** Extract a `workItemService.ts` that wraps all store operations. Route handlers should call `workItemService.findById(id)` not `store.findById(id)`. This is a structural refactor — route to TheFixer.
- **Cross-ref:** TheFixer pipeline candidate.

### QO-003: Missing Prometheus Histogram — `dependencyCheckDuration`
- **Severity:** P2
- **Category:** spec-drift / untested
- **File:** `Source/Backend/src/metrics.ts`
- **Detail:** `FR-dependency-metrics` acceptance criteria explicitly requires 4 metrics: `dependencyOperations` counter ✅, `dispatchGatingEvents` counter ✅, `dependencyCheckDuration` **histogram** ❌, `cycleDetectionEvents` counter ✅. The histogram is completely absent from `metrics.ts`. The traceability comment `// Verifies: FR-dependency-metrics` appears on the counter definitions only, masking the gap.
- **Recommendation:** Add a `Histogram` from `prom-client` for `dependency_check_duration_seconds` with appropriate buckets (e.g., `[0.001, 0.005, 0.01, 0.05, 0.1]`), instrument it around `detectCycle` and `hasUnresolvedBlockers` calls in `dependency.ts`.
- **Cross-ref:** Route to TheFixer.

### QO-004: OpenTelemetry Not Implemented
- **Severity:** P2
- **Category:** architecture-violation
- **File:** `Source/Backend/src/` (no OTel files exist)
- **Detail:** CLAUDE.md is unambiguous: "Use OpenTelemetry for distributed tracing. Auto-instrument HTTP, database, and framework calls. Add custom spans for critical paths. Propagate W3C `traceparent` header across service boundaries." Zero `@opentelemetry` packages are imported or configured anywhere in `Source/`. The logger claims to inject trace/span IDs but there is no OTel context from which to extract them. `workflow-engine.md` does not call out OTel, but CLAUDE.md states it as a non-negotiable architecture rule for "all new code."
- **Recommendation:** Add `@opentelemetry/sdk-node`, `@opentelemetry/auto-instrumentations-node`, and a `tracing.ts` bootstrap file. Instrument express HTTP routes and log `traceId`/`spanId` in log entries. Route to TheFixer.
- **Cross-ref:** Escalate to TheGuardians for any security trace propagation review.

### QO-005: `pending_dependencies` Status — Spec vs Implementation Divergence
- **Severity:** P3
- **Category:** spec-drift
- **File:** `Source/Shared/types/workflow.ts`, `Source/Backend/src/routes/workflow.ts`
- **Detail:** `FR-dependency-dispatch-gating` acceptance criteria states: "PATCH to approved with unresolved blocker → 200 with status=`pending_dependencies`". The implementation at `/dispatch` instead returns **400** with `{ error: 'Cannot dispatch: work item has unresolved blocking dependencies' }`. There is no `pending_dependencies` value in the `WorkItemStatus` enum. The VALID_STATUS_TRANSITIONS map does not include this status. The spec requirement for a soft-block with a distinct status (not a hard rejection) is not met — the system hard-blocks with an error.
- **Recommendation:** Either (a) add `PendingDependencies = 'pending-dependencies'` to `WorkItemStatus`, update VALID_STATUS_TRANSITIONS, and change the dispatch logic to set this status instead of returning 400; or (b) update the spec acceptance criteria to reflect the current 400 behavior. Decision is a product call — route to requirements-reviewer.
- **Cross-ref:** Also affects BlockedBadge component which checks `status='pending_dependencies'` that is never actually set.

### QO-006: `BlockedBadge` Amber State Unreachable
- **Severity:** P3
- **Category:** spec-drift / untested
- **File:** `Source/Frontend/src/components/BlockedBadge.tsx`
- **Detail:** `FR-dependency-blocked-badge` specifies: "amber badge for `status='pending_dependencies'`". The component renders an amber badge when `status === 'pending_dependencies'`. But per QO-005, this status is never set on any work item — the WorkItemStatus enum doesn't include it. The amber path in BlockedBadge is dead code — it will never render in production, and tests cannot exercise a path that real data never reaches.
- **Recommendation:** Resolve together with QO-005. If `pending_dependencies` status is added, ensure at least one test exercises the amber path with a real status value.

### QO-008: Two Logger Modules — Confusing Architecture
- **Severity:** P3
- **Category:** architecture-violation
- **Files:**
  - `Source/Backend/src/logger.ts` — compatibility wrapper (`export default logger`)
  - `Source/Backend/src/utils/logger.ts` — actual structured logger
- **Detail:** Two logger files exist with the same Verifies comment (`FR-WF-013`). The top-level `logger.ts` is documented as a compatibility shim for "backend-coder-2's workflow routes." Both files are production code and were recently modified. Any new developer (or agent) faces ambiguity about which to import. This also means two import paths exist that provide different interface shapes.
- **Recommendation:** Consolidate to one canonical logger. Either promote `utils/logger.ts` as the canonical import and update all imports, or make the wrapper the only public interface and re-export cleanly. Add an ESLint rule or comment discouraging direct imports of the internal module.

### QO-009: `eslint-disable` Without Justification Comments
- **Severity:** P3
- **Category:** pattern-violation
- **Files:**
  - `Source/Frontend/src/components/DependencyPicker.tsx:82` — `// eslint-disable-next-line react-hooks/exhaustive-deps`
  - `Source/Frontend/src/hooks/useWorkItems.ts:63` — `// eslint-disable-next-line react-hooks/exhaustive-deps`
- **Detail:** CLAUDE.md architecture rule: "Never swallow errors silently" — the same spirit applies to lint suppressors. Both eslint-disable lines lack an inline explanation of *why* the deps are intentionally omitted. This is a maintenance hazard: future developers cannot distinguish "deliberately excluded" from "forgotten".
- **Recommendation:** Add inline comments explaining the intention, e.g.:
  `// eslint-disable-next-line react-hooks/exhaustive-deps -- debounce reset is intentional, adding searchTerm would cause infinite loop`
