## Findings

### QO-001: 40 FR IDs in portal code have no canonical Specification
- **Severity:** P1
- **Category:** spec-drift
- **Files:**
  - `portal/Frontend/src/**` — references FR-070..FR-095, FR-DUP-01..FR-DUP-13, FR-0001
  - `portal/Backend/src/**` — same range
- **Detail:** Architecture rule states: *"Specs are source of truth — implementation traces to specs, never the other way around."*  
  The portal implementation references 40 requirement IDs that exist only in Plans files (`Plans/image-upload/requirements.md`, `Plans/orchestrator-cycle-dashboard/requirements.md`, `Plans/duplicate-deprecated-status/requirements.md`) and have never been back-ported to `Specifications/dev-workflow-platform.md`.  
  
  Breakdown:
  - `FR-070..FR-095` (26 IDs) — image upload and orchestrator dashboard features; in plans only
  - `FR-DUP-01..FR-DUP-13` (13 IDs) — duplicate/deprecated status; in plans only
  - `FR-0001` (1 ID) — zero-padded variant found in dependency tests; format inconsistency
  
  This means the canonical spec file stops at FR-069 while the actual implemented product extends significantly beyond it. Any new agent reading only `Specifications/` would have an incomplete picture of the system.
- **Recommendation:** Back-port FR-070..FR-095 and FR-DUP-01..FR-DUP-13 into `Specifications/dev-workflow-platform.md` with full description, acceptance criteria, and layer tags. Normalize `FR-0001` to `FR-001`. Route to TheFixer for the spec update (no code changes needed).
- **Cross-ref:** requirements-reviewer, TheFixer

### QO-002: FR-dependency-api-types — STILL OPEN
- **Severity:** P2
- **Category:** spec-drift / untested
- **File:** `portal/Shared/api.ts:32-59`, `portal/Frontend/src/components/shared/DependencyPicker.tsx:291,293`
- **Detail:** `UpdateBugInput` and `UpdateFeatureRequestInput` in `portal/Shared/api.ts` do not include `blocked_by?: string[]`. Requirement FR-dependency-api-types mandates this field for type-safe PATCH bodies. The frontend DependencyPicker works around the missing type with `as any` casts on lines 291 and 293 — violating TypeScript type safety end-to-end.
  
  Confirmed current state:
  ```ts
  // portal/Shared/api.ts — UpdateBugInput (line ~59)
  export interface UpdateBugInput {
    // ... no blocked_by field
  }
  
  // DependencyPicker.tsx:291
  await bugs.update(itemId, { blocked_by: blockerIds } as any);
  ```
- **Recommendation:** Add `blocked_by?: string[]` to both `UpdateBugInput` and `UpdateFeatureRequestInput` in `portal/Shared/api.ts`. Remove the two `as any` casts in `DependencyPicker.tsx`.
- **Cross-ref:** TheFixer (backend-coder/api-contract)

### QO-005: Architecture violation — Direct SQL in portal/Backend route handler
- **Severity:** P2
- **Category:** architecture-violation
- **File:** `portal/Backend/src/routes/teamDispatches.ts:37-44, 71-75`
- **Detail:** `teamDispatches.ts` executes raw SQL queries directly inside route handlers — bypassing the service layer entirely. This violates the CLAUDE.md rule: *"No direct DB calls from route handlers — use the service layer."*  
  
  ```ts
  // teamDispatches.ts:37
  rows = db.prepare(
    `SELECT * FROM team_dispatches WHERE team = ? ORDER BY dispatched_at DESC LIMIT ?`
  ).all(team, limitNum) as TeamDispatch[];
  ```
  
  Additionally, `teamDispatches.ts` has **zero `// Verifies:` traceability comments** despite being a recently added source file. All other portal route files have at minimum 2–15 Verifies comments.
  
  Note: Other portal routes (`bugs.ts`, `featureRequests.ts`, `cycles.ts`) call `getDb()` but immediately pass it to service-layer functions like `listBugs(db)` — this is the accepted pattern and not flagged here.
- **Recommendation:** Extract SQL from `teamDispatches.ts` into a `teamDispatchService.ts` service module. Add `// Verifies:` traceability pointing to the relevant spec requirement (or create one if missing).
- **Cross-ref:** TheFixer (backend-coder)

### QO-006: Traceability enforcer gate covers only 13 of 100+ requirements
- **Severity:** P2
- **Category:** spec-drift / pattern-violation
- **File:** `tools/traceability-enforcer.py` (auto-selects most recently modified `requirements.md`)
- **Detail:** The enforcer auto-selects `Plans/self-judging-workflow/requirements.md` as the "most recently modified" plan — it checks only FR-WF-001..FR-WF-013 (13 requirements). This means the verification gate that all agents must run passes silently even when:
  - portal FR-001..FR-069 lack Verifies comments
  - FR-dependency-* gaps are not caught
  - FR-DUP-* and FR-070..FR-095 are never evaluated
  
  The gate that CLAUDE.md lists as a "non-negotiable verification gate" is checking less than 10% of the codebase's requirements.
- **Recommendation:** Either: (a) add a `--all-plans` flag to the enforcer that scans all `requirements.md` files and aggregates; or (b) create a top-level `Plans/requirements-all.md` that consolidates all active FRs and always use `--file` to target it; or (c) add a second enforcer invocation in the verification gate targeting `Plans/dev-workflow-platform/requirements.md`.
- **Cross-ref:** TheFixer (tools)
