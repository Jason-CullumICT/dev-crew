## Findings

### QO-001: GET /api/search endpoint not wired into app.ts
- **Severity:** P1
- **Category:** untested / spec-drift
- **File:** `Source/Backend/src/app.ts` (missing route registration) / `Source/Backend/tests/routes/search.test.ts:1-7`
- **Detail:** `FR-dependency-search` requires a `GET /api/search?q=` cross-entity typeahead endpoint. Five tests covering this endpoint exist in `tests/routes/search.test.ts` and are marked with `// Verifies: FR-dependency-search`. The test file itself contains an explicit warning comment:
  > "NOTE: As of this review cycle the GET /api/search endpoint is NOT wired into Source/Backend/src/app.ts. These tests document the expected contract and will FAIL until the route is implemented."
  
  The `workItemsApi.searchItems()` frontend function calls `GET /api/search` (client.ts:101-104), meaning the DependencyPicker search typeahead is broken at runtime. Any `npm test` run will surface 5 failing tests.
- **Recommendation:** Implement the search route handler (filter `store.getAllItems()` by title/description, exclude deleted, return `{data: T[]}`) and register it in `app.ts` before `workItemsRouter`. The test contract in `search.test.ts` gives the full acceptance criteria.
- **Cross-ref:** TheFixer (implementation), TheGuardians (no auth concerns but confirm no unbounded list exposure risk per config threat scenario "Unbounded list exfiltration")

### QO-002: Traceability enforcer is blind to Specifications/ — 89+ requirements unmonitored
- **Severity:** P2
- **Category:** architecture-violation / spec-drift
- **File:** `tools/traceability-enforcer.py:49-57`
- **Detail:** The enforcer auto-selects the most recently modified `Plans/**/requirements.md` file. It **never scans** `Specifications/`. This means:
  - 76 requirements in `Specifications/dev-workflow-platform.md` (FR-001–069) are outside enforcement scope
  - 13 requirements in `Specifications/tiered-merge-pipeline.md` (FR-TMP-001–010) are outside enforcement scope
  - When a new plan's requirements are activated, the old plan's requirements become silently unenforced

  Running `python3 tools/traceability-enforcer.py --file Specifications/dev-workflow-platform.md` reports **76 MISSING requirements**; `--file Specifications/tiered-merge-pipeline.md` reports **13 MISSING requirements**.

  Note: FR-001–069 describe the full dev-workflow-platform product (Feature Requests, Bug Reports, Dev Cycles, etc.) which is architecturally separate from `Source/` (the self-judging workflow engine). If these specs are intentionally out-of-scope for `Source/`, the spec files should either be moved to a different location or the enforcer config in `inspector.config.yml` should explicitly acknowledge the out-of-scope corpus.
- **Recommendation:** Either (a) add a `--specs` flag to the enforcer that checks `Specifications/` against a designated implementation directory (e.g., `portal/`), or (b) add a `# SCOPE: platform-not-source` header to the Specifications that are not yet implemented, so future audits can distinguish intentional exclusion from accidental drift. Update `inspector.config.yml` to document which spec files map to which source dirs.
- **Cross-ref:** requirements-reviewer agent (owns Specifications/)

### QO-004: Missing `dependencyCheckDuration` histogram metric
- **Severity:** P2
- **Category:** spec-drift / architecture-violation
- **File:** `Source/Backend/src/metrics.ts` (missing) / `Specifications/dev-workflow-platform.md` (FR-dependency-metrics)
- **Detail:** FR-dependency-metrics specifies four Prometheus metrics:
  1. `dependencyOperations` counter ✓ (`dependency_operations_total`)
  2. `dispatchGatingEvents` counter ✓ (`dispatch_gating_events_total`)
  3. `dependencyCheckDuration` **histogram ✗ — not present in metrics.ts**
  4. `cycleDetectionEvents` counter ✓ (`cycle_detection_events_total`)

  The `dependencyCheckDuration` histogram is absent from `metrics.ts` (17 exported symbols checked). Without it, latency for readiness checks and dependency resolution is invisible to Prometheus.
- **Recommendation:** Add to `metrics.ts`:
  ```ts
  export const dependencyCheckDurationHistogram = new Histogram({
    name: 'dependency_check_duration_seconds',
    help: 'Duration of dependency readiness checks',
    labelNames: ['operation'] as const,
    registers: [registry],
  });
  ```
  Instrument `isReady()` and `computeHasUnresolvedBlockers()` in `dependency.ts`.
- **Cross-ref:** TheFixer

- **Detail:** There are two test files for each of these components at different paths with different line counts. The `tests/pages/` versions appear to be newer/more complete versions added during a later implementation pass. Both will be picked up by the Vitest test runner, causing:

- **Recommendation:** Delete the older copies in `tests/` root (check git history to confirm which is the canonical version), keeping only the `tests/pages/` versions. Verify test run passes with zero regressions after removal.
- **Cross-ref:** TheFixer

### QO-007: Two `eslint-disable-next-line` suppressions hiding potential hook issues
- **Severity:** P3
- **Category:** pattern-violation
- **File:** `Source/Frontend/src/components/DependencyPicker.tsx:82` and `Source/Frontend/src/hooks/useWorkItems.ts:63`
- **Detail:** Both suppress `react-hooks/exhaustive-deps`. In `DependencyPicker.tsx`, this could hide a stale closure over search state. In `useWorkItems.ts`, this suppresses a dependency warning on a data-fetching effect. These are common candidates for subtle infinite-loop or stale-data bugs.
- **Recommendation:** Investigate each suppression — prefer fixing the hook dependency correctly (e.g., wrapping callbacks with `useCallback`, using a `ref` for stable references) over suppressing the rule. If suppression is genuinely necessary, add an inline comment explaining why (e.g., `// intentionally omitted: adding X would cause infinite refetch`).
