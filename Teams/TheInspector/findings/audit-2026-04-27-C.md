### QO-001: Search Endpoint Unimplemented — Failing Tests Exist
- **Severity:** P1
- **Category:** spec-drift / untested
- **File:** `Source/Backend/src/app.ts` (route not wired), `Source/Backend/tests/routes/search.test.ts`
- **Detail:** `FR-dependency-search` requires `GET /api/search?q=` for the DependencyPicker typeahead. The route is absent from `app.ts`. The test file (`search.test.ts`, 5 test cases) documents this explicitly at line 1–6:
  ```
  // NOTE: As of this review cycle the GET /api/search endpoint is NOT wired into
  // Source/Backend/src/app.ts. These tests document the expected contract and will
  // FAIL until the route is implemented.
  ```
  The `DependencyPicker` component calls `searchItems()` (`api/client.ts:100`) which hits this missing route. In production the typeahead will 404 on every keystroke.
- **Recommendation:** Add a `/api/search` route to `app.ts` backed by `store.findAll()` with text filter on `title`/`description`. Wire up before merge. The tests already define the contract — passing them is the acceptance criterion.
- **Cross-ref:** TheFixer (implementation gap); FR-dependency-search, FR-dependency-picker

### QO-005: `pending_dependencies` Status Not in WorkItemStatus Enum
- **Severity:** P2
- **Category:** spec-drift / architecture-violation
- **File:** `Source/Shared/types/workflow.ts:4–15`
- **Detail:** `FR-dependency-dispatch-gating` specifies: "PATCH to approved with unresolved blocker → 200 with status=`pending_dependencies`". `Source/Shared/api-contracts.md` (line 181) documents the same behavior: "If transitioning to `approved` and `blockedBy` contains any unresolved items → set status to `pending_dependencies` instead." However `WorkItemStatus` enum has no `PendingDependencies = 'pending-dependencies'` value. The `VALID_STATUS_TRANSITIONS` has a comment at line 213 acknowledging the intent but the type is missing. As a result, dispatch gating silently blocks via 400 without ever setting `pending_dependencies` status — items stay `approved` with a 400 error rather than transitioning to `pending_dependencies`. This means cascade auto-dispatch (`onItemResolved`) cannot correctly identify and unblock items in `pending_dependencies` state.
- **Recommendation:** Add `PendingDependencies = 'pending-dependencies'` to `WorkItemStatus`. Update `VALID_STATUS_TRANSITIONS` to include `Approved → PendingDependencies` and `PendingDependencies → Approved`. Update dispatch route to set `pending_dependencies` on blocked dispatch. Update `onItemResolved` to query items in that status. Update `StatusBadge.tsx`. Run full test suite.
- **Cross-ref:** [ESCALATE → TheFixer]; api-contract agent for shared type update; FR-dependency-dispatch-gating

### QO-008: Logger Missing Development Mode Pretty-Print
- **Severity:** P3
- **Category:** spec-drift
- **File:** `Source/Backend/src/utils/logger.ts`
- **Detail:** `FR-003` and `FR-WF-013` both specify "structured JSON logging in production, pretty-printing in development." The implementation at `utils/logger.ts` unconditionally writes JSON via `process.stdout.write(JSON.stringify(entry) + '\n')` with no `NODE_ENV` check. In development this produces minified JSON rather than human-readable output, degrading developer experience and violating the spec requirement.
- **Recommendation:** Check `process.env.NODE_ENV !== 'production'` and use a formatted output path (e.g., `console.log` with colour/indentation using `JSON.stringify(entry, null, 2)` or a chalk-based formatter) for development. JSON output remains default for production. Update the acceptance criteria test in `utils/logger` tests to cover both modes.

### QO-009: Dual Logger Abstraction — Technical Debt
- **Severity:** P3
- **Category:** pattern-violation
- **Files:** `Source/Backend/src/logger.ts`, `Source/Backend/src/utils/logger.ts`
- **Detail:** Two logger modules co-exist:
  - `src/utils/logger.ts` — the actual implementation (named export `logger`)
  - `src/logger.ts` — a compatibility wrapper that re-exports `utils/logger` with a different call signature (accepts object `{msg, ...ctx}` in addition to `(string, ctx)`)
  Route files import `logger` from `../logger` (wrapper); the store imports `{logger}` from `../utils/logger` (direct). This creates two call-site conventions (object-style vs positional-style) for the same underlying logger, increasing cognitive load and risk of misuse.
- **Recommendation:** Pick one convention, update all callers to use it, delete the compatibility shim. Prefer the object-style `{msg, ...ctx}` as it produces richer structured logs. Alternatively, update `utils/logger` to accept both call forms and delete `logger.ts`.
