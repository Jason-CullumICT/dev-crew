## QO-001: `GET /api/search` Not Wired — 5 Tests Failing

- **Severity:** P1
- **Category:** spec-drift / untested
- **File:** `Source/Backend/src/app.ts` (missing route mount) + `Source/Backend/tests/routes/search.test.ts` (5 failing tests)
- **Detail:** `FR-dependency-search` requires a cross-entity search endpoint at `GET /api/search?q=`. The test file `search.test.ts` explicitly documents: *"As of this review cycle the GET /api/search endpoint is NOT wired into Source/Backend/src/app.ts. These tests document the expected contract and will FAIL until the route is implemented."*
  All 5 tests return HTTP 404 (route not mounted). The `DependencyPicker` component in the frontend calls this endpoint for typeahead; without it, the picker cannot search for items to link as dependencies.
- **Test failures (confirmed):**
  - `returns 200 and {data:[...]} with items matching the query` → Expected 200, Received 404
  - `returns empty data array when query is empty` → Expected 200, Received 404
  - `returns results from multiple work item types` → Expected 200, Received 404
  - `returns items whose description matches the query` → Expected 200, Received 404
  - `excludes deleted work items from search results` → Expected 200, Received 404
- **Recommendation:** Wire a search route in `app.ts`:
  ```typescript
  // In app.ts, add before errorHandler:
  app.get('/api/search', (req, res) => {
    const q = (req.query.q as string || '').toLowerCase();
    const all = store.getAllItems();
    const results = q ? all.filter(i =>
      i.title.toLowerCase().includes(q) ||
      i.description.toLowerCase().includes(q)
    ) : [];
    res.json({ data: results });
  });
  ```
  Or extract into `Source/Backend/src/routes/search.ts` and mount at `/api/search`.
- **Cross-ref:** TheFixer (code fix), the DependencyPicker frontend component depends on this

## QO-003: `dependencyCheckDuration` Histogram Missing from Metrics

- **Severity:** P2
- **Category:** spec-drift
- **File:** `Source/Backend/src/metrics.ts`
- **Detail:** `FR-dependency-metrics` specifies four Prometheus instruments:

  3. `dependencyCheckDuration` **histogram** ❌ (absent)

  The histogram is missing from both `metrics.ts` (not declared) and from `tests/routes/metrics.test.ts` (not tested). The spec explicitly calls for a histogram to measure dependency check latency. Without it, SLO monitoring for dependency evaluation performance is impossible.
- **Recommendation:** Add to `metrics.ts`:

  ```typescript
    buckets: [0.001, 0.005, 0.01, 0.05, 0.1, 0.5],
    registers: [registry],
  });
  ```
  Then wrap `isReady()` and `computeHasUnresolvedBlockers()` calls with a timer. Add a test in `metrics.test.ts` verifying `dependency_check_duration_seconds` appears at `GET /metrics`.

## QO-007: Dual Logger Abstraction in Source/Backend

- **Severity:** P3
- **Category:** architecture-violation / doc-stale
- **File:** `Source/Backend/src/logger.ts` (wrapper) + `Source/Backend/src/utils/logger.ts` (canonical)
- **Detail:** Two logger implementations exist side-by-side. The canonical logger at `utils/logger.ts` uses `(msg: string, ctx?: Record<string, unknown>)` signature. A compatibility wrapper at `src/logger.ts` (not in utils/) normalizes calls so routes importing `logger` as default can call it with either a string or an object `{ msg, ...rest }`.
  
  CLAUDE.md architecture rule: *"Use the project's logger abstraction, never console.log or log.Printf"* — but with two implementations, "the project's logger abstraction" is ambiguous. The `workItems.ts` route calls `logger.info({ msg: 'Work item created', ... })` while `workItemStore.ts` calls `logger.info('Work item created', { id })`. Both work due to the wrapper, but this creates an inconsistent logging call pattern across the codebase.
- **Recommendation:** Standardize on one call convention. Either (a) update `utils/logger.ts` to accept both call forms natively and remove the wrapper, or (b) update all callers to use the same pattern and remove the wrapper.
- **Cross-ref:** TheFixer (refactor)

## QO-008: `eslint-disable` in Production Frontend Code Without Explanation

- **Severity:** P3
- **Category:** pattern-violation
- **Files:**
  - `Source/Frontend/src/components/DependencyPicker.tsx:82` — `// eslint-disable-next-line react-hooks/exhaustive-deps`
  - `Source/Frontend/src/hooks/useWorkItems.ts:63` — `// eslint-disable-next-line react-hooks/exhaustive-deps`
- **Detail:** Architecture rules flag `eslint-disable` as a pattern to surface. Both suppressions silence `react-hooks/exhaustive-deps` with no explanation of why the dependency was intentionally omitted. This can mask stale-closure bugs. The pattern is likely intentional (avoiding infinite re-render loops), but without documentation the next developer cannot tell if this is safe or a latent bug.
- **Recommendation:** Add an explanatory comment before each suppression explaining WHY the dependency is excluded, e.g.:
  ```typescript
  // eslint-disable-next-line react-hooks/exhaustive-deps
  // search() is intentionally omitted — it is a stable callback ref and
  // including it would cause infinite re-fetch on every keystroke.
  ```
  Alternatively, wrap the callback with `useCallback` with the correct dependencies.
- **Cross-ref:** TheFixer (frontend cleanup)
