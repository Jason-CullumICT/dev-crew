### QO-001: GET /api/search route not registered in app.ts
- **Severity:** P1
- **Category:** spec-drift / untested (failing tests deliberately document the gap)
- **File:** `Source/Backend/src/app.ts` (missing route mount) + `Source/Backend/tests/routes/search.test.ts:1-6`
- **Detail:** `FR-dependency-search` requires a cross-entity typeahead search endpoint. The test file `search.test.ts` explicitly notes: *"NOTE: As of this review cycle the GET /api/search endpoint is NOT wired into Source/Backend/src/app.ts. These tests document the expected contract and will FAIL until the route is implemented."* The `DependencyPicker` component calls `searchItems()` which hits `GET /api/search` — the typeahead is completely non-functional in production.
- **Recommendation:** Implement the search route handler in `Source/Backend/src/routes/` and register it in `app.ts`:
  ```ts
  app.use('/api/search', searchRouter);
  ```
  The search logic should query `store.findAll()` and filter by title/description match on the `?q=` parameter.
- **Cross-ref:** [ESCALATE → TheFixer] Fix team should implement the route. Frontend DependencyPicker already wired.

### QO-002: OTel tracing absent from Source/Backend
- **Severity:** P2
- **Category:** architecture-violation / spec-drift
- **File:** `Source/Backend/src/` (no OTel imports anywhere in src/)
- **Detail:** Two separate requirements mandate OpenTelemetry tracing:
  1. `FR-WF-013` (Plans/self-judging-workflow/requirements.md): *"OpenTelemetry tracing: instrument HTTP routes and database calls; propagate W3C traceparent header"*
  2. CLAUDE.md Architecture Rule: *"Use OpenTelemetry for distributed tracing. Auto-instrument HTTP, database, and framework calls. Add custom spans for critical paths. Propagate W3C traceparent header across service boundaries."*
  
  `@opentelemetry/api` appears only in `package-lock.json` as a transitive dep. No instrumentation is initialized, no spans are created, no `traceparent` header is forwarded in any request handler.
- **Recommendation:** Add `@opentelemetry/sdk-node` and `@opentelemetry/auto-instrumentations-node` packages, create `src/tracing.ts`, initialize before app startup, add custom spans for `routeWorkItem`, `assessWorkItem`, `dispatch`, and dependency resolution operations.
- **Cross-ref:** [ESCALATE → TheFixer] Backend implementation task.

### QO-007: FR-dependency-dispatch-gating implementation deviates from spec (no pending_dependencies status)
- **Severity:** P2
- **Category:** spec-drift / architecture-violation
- **File:** `Source/Shared/types/workflow.ts` + `Source/Backend/src/routes/workflow.ts:230-244`
- **Detail:** `FR-dependency-dispatch-gating` specifies: *"Status transition to approved with unresolved blockers → response status is `pending_dependencies`"* and *"resolving blocker → dependent auto-advances from `pending_dependencies` to `approved`."* 
  
  The Source/ implementation takes a different approach: items remain in `approved` state but the `/dispatch` endpoint returns HTTP 400 when unresolved blockers exist. There is no `pending_dependencies` status in `WorkItemStatus`. The cascade correctly auto-dispatches dependents but without a `pending_dependencies` state, items blocked before dispatch remain visually as `approved` in the UI with no indication they're waiting on dependencies.
  
  The spec comment in `Source/Shared/types/workflow.ts:214` says "Support for pending_dependencies blocking" but the status itself was never added to the enum.
- **Recommendation:** Either: (a) add `PendingDependencies = 'pending-dependencies'` to `WorkItemStatus`, update `VALID_STATUS_TRANSITIONS`, and transition items to this status when blockers block the approve step; or (b) formally document this as an intentional design deviation in the spec and update `workflow-engine.md`.
- **Cross-ref:** [ESCALATE → TheFixer] cross-layer: Shared/types + Backend + Frontend.

- **Category:** pattern-violation

- **Detail:** Two `// eslint-disable-next-line react-hooks/exhaustive-deps` suppressions exist in production source.
