# ABAC Comprehensive Expansion — Design Specification

> **For agentic workers:** This spec covers 6 phases. Each phase should be implemented as a separate plan using superpowers:writing-plans. Do not attempt to implement all phases at once.

**Goal:** Expand the existing ABAC access control demo from a policy configuration tool into a comprehensive security operations platform covering real-time monitoring, hardware I/O, response automation, advanced access rules, visitor management, and external integrations.

**Architecture:** Front-end only (React 19 + Zustand). All data lives in browser state with localStorage persistence. No backend. Events are generated by a built-in simulation engine with manual trigger overrides.

**Phasing:** SOC-First — fill the biggest gap (real-time monitoring) first, then hardware, then rules, then advanced features.

## Phase 1 — SOC Monitoring

**Goal:** Add real-time event streaming, alarm management, and a Security Operations Center view. Make the demo feel alive.

### 1.1 New Types

```typescript
type EventSeverity = 'critical' | 'warning' | 'info'
type EventCategory = 'access' | 'alarm' | 'system' | 'intrusion'
type AlarmState = 'active' | 'acknowledged' | 'escalated' | 'cleared'
type ThreatLevel = 'normal' | 'elevated' | 'high' | 'critical' | 'lockdown'

interface SecurityEvent {
  id: string
  timestamp: string              // ISO 8601
  category: EventCategory
  severity: EventSeverity
  eventType: string              // 'access_granted' | 'access_denied' | 'door_forced' | 'door_held' |
                                 // 'sensor_trip' | 'controller_offline' | 'arm_state_change' | 'panic_button'
  siteId: string
  zoneId?: string
  doorId?: string
  userId?: string
  message: string
  metadata: Record<string, string>
}

interface Alarm {
  id: string
  triggerEventId: string
  severity: EventSeverity
  state: AlarmState
  siteId: string
  zoneId?: string
  doorId?: string
  title: string
  acknowledgedBy?: string        // userId
  acknowledgedAt?: string
  escalatedAt?: string
  clearedAt?: string
  notes: string[]
}
```

### 1.3 Event Simulator Behaviour

**Generation rate by speed setting:**
- `0` (paused): No auto-generation. Manual triggers only.
- `1` (1x real-time): ~1 event every 2-5 seconds. Realistic for a mid-size campus.
- `10` (10x fast-forward): ~1 event every 200-500ms. For demos showing volume.

**Event probability weights — adjusted by time-of-day and arm state:**

| Condition | access_granted | access_denied | door_forced/held | sensor_trip | system | arm_change |
|-----------|---------------|---------------|-----------------|-------------|--------|------------|
| Business hours + disarmed | 80% | 10% | 3% | 2% | 5% | 0% |
| After hours + armed | 20% | 30% | 5% | 20% | 10% | 15% |
| Lockdown | 5% (override only) | 60% | 5% | 15% | 10% | 5% |

**Smart behaviour:**
- Events reference real users, doors, and grants from the store — not random strings.
- Access granted events run through the actual ABAC engine to verify the grant would work.
- Access denied events pick a user who genuinely lacks permission for the target door.
- Burst patterns: site arm/disarm generates a cluster of 5-10 events in quick succession.

**Alarm-generating events (auto-create Alarm from SecurityEvent):**

| Event Type | Alarm Severity | Alarm Title Pattern |
|-----------|---------------|---------------------|
| `door_forced` | critical | "Door Forced Open — {doorName}" |
| `door_held` (>30s) | warning | "Door Held Open — {doorName}" |
| `sensor_trip` (armed zone) | critical | "Sensor Trip — {zoneName}" |
| `access_denied` (3+ in 60s, same door) | warning | "Repeated Denied Access — {doorName}" |
| `controller_offline` | warning | "Controller Offline — {controllerName}" |
| `panic_button` | critical | "Panic Button — {location}" |

### 1.4 Scenario Presets

Each preset is a scripted sequence of SecurityEvents injected over a time window.

| Preset | Description | Duration | Events |
|--------|-------------|----------|--------|
| **After-Hours Breach** | Door forced at Server Room → alarm → 3 denied access attempts → escalation | 30s | ~8 events |
| **Fire Evacuation** | Fire alarm trigger → all doors in zone unlock → evacuation events → muster gap report | 45s | ~15 events |
| **Normal Business Day** | Steady access granted stream → occasional denied → one door held open → auto-clear | 60s | ~30 events |
| **Full Lockdown** | Trigger site lockdown → all zones arm → deny all access → only override users enter | 20s | ~10 events |
| **Shift Change (peak traffic)** | Burst of 50+ access events over 2 minutes simulating end-of-day transition, schedule window changeover, zone re-arming | 120s | ~50 events |
| **Custom Event** | Pick site → door → event type → severity → trigger manually | instant | 1 event |

### 1.5 Monitor Page Layout (Command Center)

**Top bar:** SeverityBar — 4 cards showing Critical count, Warning count, Info count, Sites OK (green/total).

**Main area — 2/3 + 1/3 split:**
- Left (2/3): **EventFeed** — scrolling list of SecurityEvents. Newest at top. Auto-scrolls unless user has scrolled up (pause indicator). Filterable by severity, category, site. Each row shows: severity dot, timestamp, message, site tag, user name if applicable.
- Right (1/3): **Alarm Queue** — active/acknowledged alarms sorted by severity then time. Each AlarmCard shows: title, severity badge, site, time since trigger, current state, ACK/Escalate/Clear buttons, notes field.

**Scenario trigger:** Floating action button (bottom-right) opens the ScenarioPanel slide-out drawer.

### 1.6 Dashboard Integration

- "Sites in Alarm" stat card pulses red border when count > 0.
- "Recent Events" section replaces the current intrusion-only timeline with the last 10 SecurityEvents (uses the same EventFeed component in compact mode).
- New "Active Alarms" section shows up to 3 AlarmCards. "View all in Monitor →" link.

### 1.7 Header Bar Integration

- **SimulationToggle:** ▶/⏸ button + speed dropdown (1x / 10x) + green "LIVE" indicator when running. Positioned left of the ⌘K trigger.
- **Alarm badge:** Red circle with unacknowledged alarm count overlaid on the Monitor sidebar icon. Hidden when count is 0.

### 1.8 Store Additions

```typescript
// New state
events: SecurityEvent[]           // capped at 500, newest first
alarms: Alarm[]                   // persists until cleared (no cap)
simulationSpeed: 0 | 1 | 10      // 0 = paused

// New actions
addEvent: (event: SecurityEvent) => void
addAlarm: (alarm: Alarm) => void
acknowledgeAlarm: (id: string, userId: string) => void
escalateAlarm: (id: string) => void

```

Events and alarms should be excluded from localStorage persistence (ephemeral — regenerated by simulator on each session). simulationSpeed persists.

### 1.9 Sidebar Navigation

New "Monitor" entry in the primary nav section (between Canvas and Oracle). Icon: `Monitor` from lucide-react. Color: `#ef4444` (red-500). Red dot badge when unacknowledged alarms > 0.

## Phase 2 — Hardware I/O Model

**Goal:** Extend the minimal door model into a full device tree. Every door becomes a container with attached input and output devices, each mapped to controller I/O ports.

### 2.1 New Types

```typescript
type InputDeviceType = 'card_reader' | 'rex_button' | 'door_contact' | 'pir_sensor' |
                       'glass_break' | 'panic_button' | 'intercom'

type OutputDeviceType = 'electric_strike' | 'mag_lock' | 'siren' | 'strobe' |
                        'camera_trigger' | 'relay_output'

type DeviceStatus = 'online' | 'offline' | 'tamper' | 'fault' | 'low_battery'

interface InputDevice {
  id: string
  name: string
  type: InputDeviceType
  doorId: string
  controllerId: string
  port: number                   // controller I/O port number
  status: DeviceStatus
  config: Record<string, string> // type-specific config (e.g., wiegandBits: '26')
}

interface OutputDevice {
  id: string
  name: string
  type: OutputDeviceType
  doorId?: string                // optional — sirens/strobes may be zone-level, not door-level
  zoneId?: string
  controllerId: string
  port: number
  status: DeviceStatus
  config: Record<string, string> // type-specific (e.g., lockType: 'fail-safe', holdTime: '5')
}
```

### 2.2 New UI

- **Door Configuration page:** Visual "wiring diagram" view. Select a door → see its attached readers, sensors, lock, and camera trigger as a schematic. Click a device to configure it. Shows controller port assignments.
- **Hardware Inventory page:** Flat table of all InputDevice + OutputDevice across all sites. Columns: name, type, door, site, controller, port, status. Filterable by type, status, site. Status badges pulse for offline/fault.
- **Canvas integration:** New compact node types for readers (small circle with reader icon) and sensors (small triangle), connected to their parent door node via thin dotted lines.

### 2.3 Seed Data

Each door in the seed gets a default hardware kit:
- 1 card reader (outdoor side)
- 1 REX button (indoor side)
- 1 door contact sensor
- 1 electric strike (or mag lock for secure zones)
- Secure/Restricted doors also get: 1 PIR sensor, 1 camera trigger

Zone-level devices (not door-specific):
- Each Perimeter zone: 2 PIR sensors, 1 siren, 1 strobe
- Each Secure/Restricted zone: 1 glass break sensor, 1 siren, 1 strobe

### 2.4 Event Simulator Integration

Phase 1's event simulator gains richer events:
- `door_contact_open` / `door_contact_close` events tied to the door contact device
- `reader_tamper` events for card readers
- `device_offline` / `device_online` events per device (not just per controller)
- `pir_trigger` events in armed zones

## Phase 3 — Response Rules Engine

**Goal:** Configurable event-to-action mapping. "When X happens at conditions Y → do Z." Admins build rules; the simulator evaluates them.

### 3.1 Rule Structure

```typescript
interface ResponseRule {
  id: string
  name: string
  enabled: boolean
  priority: number               // lower = higher priority (evaluated first)
  trigger: {
    eventTypes: string[]         // e.g., ['door_forced', 'sensor_trip']
    severities?: EventSeverity[] // filter to specific severities
  }
  conditions: {
    siteIds?: string[]           // empty = all sites
    zoneTypes?: ZoneType[]       // e.g., ['Restricted', 'Secure']
    armStates?: ZoneStatus[]     // e.g., ['Armed']
    timeWindow?: string          // scheduleId — rule only active during this schedule
    threatLevel?: ThreatLevel[]  // e.g., ['elevated', 'high']
  }
  actions: ResponseAction[]
}

type ThreatLevel = 'normal' | 'elevated' | 'high' | 'critical' | 'lockdown'

interface ResponseAction {
  type: 'lock_door' | 'lock_zone' | 'lock_site' | 'unlock_door' | 'unlock_zone' |
        'activate_siren' | 'activate_strobe' | 'trigger_camera' |
        'send_notification' | 'escalate_alarm' | 'change_threat_level' |
        'arm_zone' | 'disarm_zone'
  targetId?: string              // doorId, zoneId, or siteId depending on type
  params: Record<string, string> // action-specific (e.g., notifyEmail, cameraPreset)
}
```

### 3.2 Threat Levels

Global system state affecting all response rules.

| Level | Description | Typical Actions |
|-------|-------------|----------------|
| Normal | Standard operations | No additional restrictions |
| Elevated | Heightened awareness | Extra logging, increased patrol |
| High | Active concern | Arm all perimeter zones, deny visitors |
| Critical | Active threat | Lock all secure zones, deny all non-essential access |
| Lockdown | Full lockdown | Lock everything, only override-clearance users can enter |

Threat level is a global store value. Response rules can reference it as a condition. Threat level changes are logged as SecurityEvents.

### 3.3 Escalation Chains

```typescript
interface EscalationChain {
  id: string
  name: string
  steps: EscalationStep[]
}

interface EscalationStep {
  delayMinutes: number           // time after alarm trigger before this step fires
  notifyUserIds: string[]        // who to notify
  autoActions: ResponseAction[]  // what to do automatically
}
```

### 3.4 New UI

- **Rules Builder page:** Visual WHEN → AND → THEN editor. Trigger dropdown, condition chips, action cards. Drag to reorder priority.
- **Threat Level indicator:** Pill in the header bar showing current level with color coding. Click to change (requires permission).
- **Escalation Chain editor:** Define chain name, add steps with delay + notify + actions.

## Phase 4 — Advanced Access Control

**Goal:** Security features beyond basic badge-in/badge-out.

### 4.1 Features

| Feature | Description | Configuration |
|---------|-------------|---------------|
| **Anti-Passback** | Must badge in before badging out. Prevents card sharing. | Per-zone: hard (deny entry) or soft (log only). Timed reset (e.g., forgiven after 4 hours). |
| **Two-Person Rule** | Two authorised people must badge within 30s for door to unlock. | Per-door toggle. Both users must have the required grant. |
| **Escort Mode** | Visitors must be accompanied by an escort. Escort badges first, visitor has 15s to badge. | Per-door toggle. Escort must have 'escort' action in their grant. Visitor flagged as `type: 'visitor'`. |
| **Door Interlock (Mantrap)** | Two doors where one must close before the other opens. | Door pairs defined in config. Interlock state tracked in store. |
| **Muster / Evacuation** | Track who is "inside" each zone based on badge events. On evacuation, generate a muster report of who hasn't checked out. | Muster report page. Triggered by evacuation scenario or manual button. |
| **Time-Limited Access** | Grants with explicit start/end dates. Auto-expire. | `validFrom` and `validUntil` fields on Grant type. Engine checks these during evaluation. |

### 4.2 Access Engine Changes

- `evaluateAccess` gains anti-passback check (query last event for this user+zone).
- Two-person rule is a door-level pre-check before ABAC evaluation runs.
- Escort mode: evaluate escort's access first, then mark the door as "escort-primed" for 15s.
- Time-limited: add `validFrom`/`validUntil` to Grant, check in `collectGrants`.

### 4.3 New UI

- **Door detail panel:** Shows interlock pair, two-person rule, escort mode toggles.
- **Muster Report page:** Grid of zones showing occupancy count, list of people inside, "unaccounted" list during evacuation.
- **Anti-passback status:** Per-user indicator showing their current "in/out" state per zone.

## Phase 5 — Visitor & Credential Management

**Goal:** Complete the people side of the access control lifecycle.

### 5.1 Visitor Flow

Pre-register (host submits request) → Approve → Check-in (reception) → Issue temp credential → Assign escort → Track access events → Check-out → Expire credential.

### 5.2 Credential Types

```typescript
type CredentialType = 'proximity_card' | 'smart_card' | 'pin' | 'mobile' | 'biometric'
type CredentialStatus = 'active' | 'suspended' | 'revoked' | 'expired'

interface Credential {
  id: string
  userId: string
  type: CredentialType
  status: CredentialStatus
  cardNumber?: string            // for card types
  facilityCode?: number          // for Wiegand cards
  pin?: string                   // for PIN type (hashed in real system)
  issuedAt: string
  expiresAt?: string
  suspendedAt?: string
  revokedAt?: string
}
```

### 5.3 New UI

- **Visitor Registration page:** Form for pre-registration. Host selects visitor, dates, escort, areas.
- **Reception Kiosk view:** Simplified check-in interface. Scan/enter visitor details → issue badge → print visitor pass.
- **Credential Management:** Per-user credential list. Issue new, suspend, revoke, replace lost card (transfer grants).

## Phase 6 — Integration & Compliance

**Goal:** External system interfaces and regulatory compliance reporting.

### 6.1 Integrations (Simulated)

| Integration | Behaviour |
|-------------|-----------|
| **Fire Alarm** | Input trigger that auto-unlocks all doors in the zone, overriding arm state. Non-cancellable by access control. Generates SecurityEvents. |
| **CCTV/VMS** | Alarm events annotate with camera metadata (recording started, PTZ preset moved). Simulated as event metadata fields. |
| **Elevator** | Modelled as a special door type where "zones" are floors. User badges at elevator reader → only authorised floors are selectable. |
| **BMS (Building Management)** | HVAC lockout on alarm. Lighting override on evacuation. Simulated as response rule actions. |
| **HR Sync** | "Import" button that simulates receiving a new-starter or leaver notification from HR. Auto-provisions into groups or auto-suspends + revokes credentials. |

### 6.2 Compliance Reports

- **Access Audit Trail:** Who accessed where, when, granted/denied. Filterable by user, door, date range. Exportable.
- **Alarm Response Times:** Average time from alarm trigger to acknowledgment. By site, by severity.
- **Policy Coverage Gaps:** Doors with no policy, users with no group, grants with no schedule.
- **Who Has Access Where:** Matrix of users × doors showing effective permissions.

### 6.3 System Health Dashboard

- Controller heartbeat status (online/offline/stale).
- Device health: reader tamper, low battery, comms fault.
- Event throughput graph (events per minute over last hour).
- Storage/memory usage indicator (event buffer fill level).

## Phase Dependencies

```
Phase 1 (SOC Monitoring) — standalone, no dependencies
    ↓
Phase 2 (Hardware I/O) — extends Phase 1 events with device-level detail
    ↓
Phase 3 (Response Rules) — consumes events (P1) and devices (P2)
    ↓
Phase 4 (Advanced AC) — extends access engine, uses events (P1) for anti-passback
    ↓
Phase 5 (Visitors) — extends user model, uses credentials concept
    ↓
Phase 6 (Integrations) — consumes all prior phases
```

Phases 4 and 5 can be swapped in order. Phase 6 must be last.

---

## What Is NOT Changing

- All existing pages, engines, store entities, and canvas features remain intact.
- The ABAC evaluation engine (`accessEngine.ts`) is extended, not replaced.
- The Zustand store is extended with new slices, not restructured.
- The existing seed data (users, groups, grants, schedules, policies, doors, zones, sites, controllers) is preserved and extended with hardware device data.
- All existing UI (Dashboard, Canvas, Oracle, Reasoner, Intrusion, entity CRUD pages) continues to work. New features are additive.
