Access Control in Practice

How administrators actually configure RBAC vs ABAC in real-world software.

1. Role-Based Access Control (RBAC) Dashboard

Permissions are grouped into static "Roles". Users are assigned to these roles.

Why it's rigid: If Charlie needs to temporarily edit one specific project, you either have to change his role to Developer (giving him edit access to everything) or create a brand new custom role just for him.

2. Attribute-Based Access Control (ABAC) Policy Builder

Access is evaluated dynamically in real-time based on rules evaluating user, resource, and context attributes.

Active Policy: Secure Financial Document Access

STATUS: ENFORCED
ALLOW operations on resource type
IF User.Department Resource.Department
AND User.SecurityClearance Resource.SensitivityLevel
AND Environment.TimeOfDay AND
AND Environment.Source_IP
Why it's dynamic: It doesn't matter what the user's "Role" title is. If the CFO tries to access a confidential finance document from a coffee shop at 11:00 PM (violating the Time and IP constraints), access is automatically blocked.
